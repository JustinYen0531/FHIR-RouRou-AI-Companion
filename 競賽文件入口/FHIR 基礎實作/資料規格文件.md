## 1. 概覽

本專案使用 FHIR R4 標準，並在實作中參考了台灣核心實作指引（TW Core IG）的資源結構。本文件詳細記錄了「AI 陪伴對話」如何轉化為結構化的 FHIR 資源，以及各個欄位的具體定義與資料映射邏輯。

## 2. 核心資源與 Profile 對應

本系統產生的 FHIR Bundle 包含以下核心資源。在轉換過程中，系統會自動生成各資源間的參照關係（Reference）。

| FHIR 資源類型 | 主要用途 | 對應 TW Core / 基幹 Profile |
| --- | --- | --- |
| `Patient` | 病人基本資料（姓名、性別、出生日期、識別碼） | `TWCorePatient` |
| `Encounter` | 就醫 / 會談情境（此對話 Session 的性質、狀態與時間） | `TWCoreEncounter` |
| `Observation` | 關鍵觀察（症狀、量表得分、風險指標、功能評估） | `TWCoreObservation` |
| `Condition` | 推斷的健康狀況（憂鬱、焦慮等初步印象） | `TWCoreCondition` |
| `Composition` | 結構化臨床記錄文件（整合摘要與對話結果） | `TWCoreComposition` |
| `DocumentReference` | 指向外部文件（如對導出之 JSON 或病人審閱後的 PDF 參照） | `TWCoreDocumentReference` |

## 3. 詳細欄位映射（Mapping Logic）

### 3.1 Patient（病人）

| 欄位名稱 | 資料來源 | 說明 / 範例 |
| --- | --- | --- |
| `id` | 內部系統 UID | e.g. `patient-rourou-001` |
| `name` | Session 設定 | `text`: "魯魯" |
| `gender` | Session 設定 | `male` / `female` / `other` |
| `birthDate` | Session 設定 | `YYYY-MM-DD` |
| `identifier` | 固定測試 ID | `system`: `https://www.example.org/patients`, `value`: "P12345" |

### 3.2 Observation（觀察值 - 情緒與量表指標）

這是本系統最核心的臨床資料承載者。

| 欄位名稱 | 資料來源 | 說明 / 範例 |
| --- | --- | --- |
| `code` | LOINC / 系統自定義代碼 | e.g. `71354-5` (Depression severity) |
| `subject` | `Patient` 參照 | 指向對應的 `Patient` 資源 |
| `effectiveDateTime` | 對話時間戳記 | `ISO 8601` 格式 |
| `valueQuantity` | AI 評分結果 | 量表得分（例如 HAM-D 單項分） |
| `valueString` | AI 擷取之質性描述 | 例如：對「興趣喪失」的具體敘述 |
| `note` | 病人回饋 / 備註 | 病人審閱時手動加入的註解 |
| `interpretation` | 嚴重程度評等 | `Normal`, `Abnormal`, `Critical` |

### 3.3 Encounter（就醫會談）

| 欄位名稱 | 資料來源 | 說明 / 範例 |
| --- | --- | --- |
| `status` | 系統開發狀態 | 固定為 `finished` |
| `class` | 服務類別代碼 | `AMB` (ambulatory) |
| `subject` | `Patient` 參照 | 指向對應的 `Patient` 資源 |
| `period` | 對話起始與結束時間 | `start` 與 `end` 時間戳記 |

### 3.4 Composition（臨床摘要文件）

將所有的 `Observation` 與 `Encounter` 串聯成一份可供醫師閱讀的完整報告。

| 欄位名稱 | 資料來源 | 說明 / 範例 |
| --- | --- | --- |
| `type` | 文件類型代碼 | `11506-3` (Progress note) |
| `author` | AI 服務識別 | `Practitioner` 或 `Organization` (RouRou AI) |
| `title` | 固定標題 | "AI 診前引導與情緒狀態彙整報告" |
| `section` | 依類別分段 | 包含情緒、行為、認知、量表、風險警示等 Segment |

## 4. 資料轉換管線 (Data Transformation Pipeline)

1. **對話紀錄導出 (Session Export)**：系統提取包含對話文字、AI 分析標籤與初步分段的 JSON 資料。
2. **FHIR 建構器 (Bundle Builder)**：`fhirBundleBuilder.js` 模組讀取 export 資料，並依序建立各個 Resource 物件。
3. **主識別碼 (ID) 與參照 (Reference) 建立**：自動為每個資源生成唯一的 UUID，並建立正確的鏈結（如 `Observation.subject` 指向 `Patient`）。
4. **TW Core Profile 附加**：在 `meta.profile` 欄位加入對定義之台灣核心規格的鏈結。
5. **Bundle 組裝與驗證**：封裝成 `transaction` 或 `collection` 類型的 Bundle。
6. **本地端交付與展示**：產生可供 `fhirDeliveryServer.js` 直接調用的靜態 FHIR JSON 檔案。

## 5. 代碼系統 (Coding Systems)

為確保互通性，本專案優先採用以下國際及在地化代碼系統：

- **LOINC**：用於臨床觀察代碼（如憂鬱量表、生理指標）。
- **SNOMED CT**：用於臨床發現、症狀描述與診斷印象。
- **TW Core ValueSets**：用於性別、身分識別類型、醫療行為分類等。

## 6. 特殊處理：自定義代碼 (System Codes)

針對 AI 特定分析標籤（如 `VoidBox` 模式或特定的輔助標籤），若 LOINC 尚未定義，系統會使用 `https://rourou.ai/fhir/codesystem/tags` 作為 URI 並定義內部代碼，確保資料在內部傳遞時不失真。
