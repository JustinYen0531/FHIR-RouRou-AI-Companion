# AI Companion 新版流程圖白話版

## 一句話先講清楚
- 現在的系統不是每一句話都偷偷幫你做完整醫療摘要。
- 現在分成兩層：
  - `聊天層`：先把人接住、先回話、先做必要判斷
  - `輸出層`：只有你真的要摘要、病人稿、FHIR draft 時才生成

這樣做的原因很直接：
- 比較快
- 比較省 token
- 比較不容易撞供應商限流

## 新版整體流程
```text
使用者輸入
  -> 先看是不是模式指令
  -> 再看是不是高風險內容
  -> 再看是不是要輸出醫師摘要 / 病人稿 / FHIR
  -> 如果都不是，就走一般聊天模式
```

## 1. 模式指令流程
如果使用者輸入這些：
- `auto`
- `void`
- `soulmate`
- `mission`
- `option`
- `natural`
- `clarify`

系統不會去跑很重的模型分析，只會：
- 更新模式狀態
- 回一句確認文字

白話理解：
- 這一步像是「切換聊天風格」
- 不是「開始做完整整理」

## 2. 高風險流程
如果使用者說：
- `我想死`
- `不想活了`
- 類似自傷、自殺、立即危險的內容

系統會直接：
- 把這輪標成高風險
- 停止一般模式分流
- 直接走安全回應

白話理解：
- 先救急，不先做摘要
- 先給安全回應，不先做醫師文件

## 3. 一般聊天流程
如果不是模式指令，也不是高風險，也不是輸出指令，就走一般聊天。

這時系統做的是：
- 更新少量重要記憶
- 判斷現在比較適合哪一種聊天模式
- 產生一段回覆

### 一般聊天時會保留的記憶
- 目前模式 `active_mode`
- 是否高風險 `risk_flag`
- 有沒有待追問問題 `pending_question`
- follow-up 次數
- 最近的重要標籤
- 最近的 HAM-D 線索
- 互動負擔程度

白話理解：
- 它會記住「重要線索」
- 但不會每輪都生成「完整報告」

## 3.5 Auto 分流現在到底怎麼判斷
`Auto` 沒有被拿掉。現在只是從 Dify 畫布節點，改成程式裡的分流流程。

現在的順序是：

```text
一般輸入
  -> 先看有沒有手動指定模式
  -> 再看是不是高風險
  -> 再看互動負擔高不高
  -> 最後再看句子內容比較像哪一種意圖
```

### 第一步：如果你有明確指定模式，就先尊重你
例如你直接輸入：
- `void`
- `soulmate`
- `mission`
- `option`
- `natural`
- `clarify`

那系統不再猜，直接切到那個模式。

白話理解：
- 你有指定，就照你的意思
- `Auto` 不會硬跟你唱反調

### 第二步：如果是高風險內容，直接跳去 Safety
例如：
- `我想死`
- `不想活了`
- 類似立即危機句子

這時不做一般模式判斷，直接走安全回應。

白話理解：
- 安全優先
- 先處理危機，不先猜你要哪一種聊天風格

### 第三步：先看互動負擔高不高
系統會先做一個「互動負擔」判斷，也就是看你現在是不是：
- 很累
- 很短句
- 很難思考
- 不想被問太多

這一步會產生幾種判斷資訊：
- `burden_level`
- `response_style`
- `followup_budget`

它的作用是先決定：
- 需不需要降低對話負擔
- 需不需要少問問題
- 需不需要改成更省力的模式

### 第四步：如果負擔很高，先做低負擔分流
如果系統判斷你現在很疲憊、很短句、很不想想，會先做低能量判斷。

這一步只會先偏向三種：
- `Option`
- `Soulmate`
- `繼續 Auto`

白話理解：
- 如果你現在很累，就不要先把你丟進很複雜的整理流程
- 系統會先偏向：
  - `Option`：給你少量選項，減少思考負擔
  - `Soulmate`：先陪你，不急著逼你整理

### 第五步：如果沒有被低負擔分流攔下，才做意圖判斷
如果系統覺得你還能正常聊天，它才會進一步判斷你現在比較像哪一種意圖。

現在它會在這六種模式裡選一個：
- `Void`
- `Soulmate`
- `Mission`
- `Option`
- `Natural`
- `Clarify`

### 它實際上怎麼偏向這六個模式

#### 1. `Void`
偏向條件：
- 你明顯只想傾倒情緒
- 不想被分析
- 不想被引導
- 不想講太多，只想被接住

例子：
- `我只是想說出來`
- `不要問我`
- `我只想安靜一下`

系統理解：
- 這時不要太多結構化互動
- 先承接

#### 2. `Soulmate`
偏向條件：
- 你需要被安撫、陪伴、共感
- 不是立即危險，但情緒很重
- 你比較需要像朋友一樣的回應

例子：
- `今天真的很難受`
- `我覺得自己很糟`
- `好想有人懂我`

系統理解：
- 先給溫度
- 不急著做任務整理

#### 3. `Mission`
偏向條件：
- 你明顯想整理問題
- 想準備給醫師
- 想有結構、想往任務方向走
- 有「我要怎麼做」「幫我整理」這種目標感

例子：
- `幫我整理最近的狀況`
- `我想知道怎麼跟醫師說`
- `我想把重點列出來`

系統理解：
- 這時比較適合進任務導向、整理導向的回答

#### 4. `Option`
偏向條件：
- 你現在很累
- 你不想思考太多
- 你適合被給 2 到 3 個低負擔選項

例子：
- `不知道該怎麼辦`
- `好累`
- `隨便，你幫我選`

系統理解：
- 不要讓你自己生成很多內容
- 先用選項減輕負擔

#### 5. `Natural`
偏向條件：
- 一般正常聊天
- 沒有明顯危機
- 也沒有很強的任務意圖
- 也不需要特別低負擔或特別澄清

例子：
- `今天工作有點多`
- `最近心情有一點差`
- `不知道怎麼放鬆`

系統理解：
- 就先自然聊天
- 邊聊邊蒐集重要線索

#### 6. `Clarify`
偏向條件：
- 你的句子太模糊
- 缺關鍵資訊
- 系統如果不先補問，容易回偏

例子：
- `不知道`
- `就是很怪`
- `嗯`

系統理解：
- 先問一個最小、最省力的釐清問題
- 不是立刻展開大段分析

### 所以 Auto 的真正機制是什麼
一句話版本：

```text
Auto = 先尊重明確指令
     -> 再先處理危險
     -> 再降低高負擔互動
     -> 最後才在六種模式裡選最適合的一個
```

### 為什麼現在這樣比以前合理
以前在 Dify，你看到的是很多 classifier / gate 節點，看起來像很細，但也比較重。

現在這版把邏輯整理成：
- 明確指令優先
- 安全優先
- 低負擔優先
- 意圖分類最後做

這樣的好處是：
- 比較快
- 比較不亂跳
- 比較符合心理陪伴產品的實際優先順序

## 4. Follow-up 流程
如果前一輪系統已經留下：
- `pending_question`

那下一輪會先處理這個待追問問題。

白話理解：
- 系統不會忘記剛剛還沒問完的事
- 但追問也有限制，不會無限追著問

## 5. 六個聊天模式現在還在
現在保留的模式有：
- `Void`
- `Soulmate`
- `Mission`
- `Option`
- `Natural`
- `Clarify`

差別在於：
- 以前：很多模式後面會順手帶出一整串摘要流程
- 現在：模式主要只影響「怎麼回話」

白話理解：
- 模式決定聊天風格
- 不再代表每次都要生成文件

## 6. Mission / Option 現在怎麼跑
`Mission` 跟 `Option` 還是有檢索能力，但改成比較節制。

現在的原則是：
- 只有在相關模式下，才用本地知識資料幫助回覆
- 不會在每一次自然聊天都硬做檢索

白話理解：
- 需要知識時才查
- 不需要時不要硬查

## 7. 什麼時候才會產生重型輸出
現在只有兩種情況會跑重型輸出：

### 方法 A：按按鈕
前端按鈕：
- `整理給醫師`
- `病人審閱稿`
- `FHIR Draft`

### 方法 B：輸入明確指令
例如：
- `幫我整理給醫生`
- `產生病人審閱稿`
- `產生FHIR draft`

這時系統才會去生成：
- clinician summary
- patient review
- patient authorization
- FHIR delivery
- delivery readiness

白話理解：
- 平常聊天不自動做
- 你真的要時才做

## 8. 為什麼新版比較快
因為以前最花成本的是這一段：

```text
每輪聊天
  -> tags
  -> burden
  -> hamd
  -> summary draft
  -> clinician summary
  -> patient review
  -> authorization
  -> fhir draft
  -> delivery readiness
```

現在改成：

```text
每輪聊天
  -> 必要判斷
  -> 模式回覆
  -> 保存重要記憶

真的要輸出時
  -> 才跑 summary / patient / fhir 長鏈
```

所以新版比較快，是因為：
- 一般聊天少打很多次模型
- 重型工作延後到真正需要時才做

## 9. 現在你可以怎麼理解這套系統
你可以把它想成兩個角色：

### 角色 1：陪你聊天的人
負責：
- 接住情緒
- 分流模式
- 危險時先做安全回應
- 記住重要線索

### 角色 2：幫你出文件的人
負責：
- 醫師摘要
- 病人審閱稿
- FHIR draft

以前這兩個角色每句話都一起上班。  
現在改成：
- 平常先由聊天的人回
- 你需要時再叫文件的人出來

## 10. 現在最重要的差異
- `聊天` 和 `輸出` 被拆開了
- `記住重點` 和 `生成完整文件` 被拆開了
- `模式切換` 只是切聊天風格，不再等於生成報告
- `醫師摘要 / FHIR` 不是每句都跑，而是按需觸發

## 11. 你之後看畫面時怎麼判斷現在系統在做什麼
如果你只是正常聊天：
- 它應該很快回話
- 不應該每次都很久

如果你按了：
- `整理給醫師`
- `病人審閱稿`
- `FHIR Draft`

那這一輪比較慢是正常的，因為這時系統才會跑重型生成。

## 12. 現在這版的真正目標
- 平常聊天要穩
- 高風險要即時
- 重要記憶不能丟
- 文件輸出要保留
- 不要再因為每輪都跑太多東西，把 API 配額打爆

## 14. 高風險詞庫現在應該怎麼設計
如果要讓安全模式更穩，最好的方式不是每次都靠 AI 猜，而是先有一層可維護的高風險詞庫。

白話理解：
- **先用詞庫擋住明確危險句子**
- **只有模糊、灰色地帶的內容，再交給 AI 補判斷**

### 第一層：直接命中就進安全模式
這一層不需要 AI 猜，只要命中就直接切安全模式。

建議至少包括：

#### A. 明確自殺意念
- `我想死`
- `我想自殺`
- `我不想活了`
- `活不下去`
- `不如死一死`
- `結束生命`
- `想消失`
- `如果消失就好了`

#### B. 明確自傷行為
- `我割了自己的手臂`
- `我割腕`
- `我拿刀割自己`
- `我用刀劃自己`
- `我自殘`
- `我傷害自己`
- `我把自己弄傷`

#### C. 已發生傷害 / 立即危險
- `我流了很多血`
- `我現在在流血`
- `我剛吞藥`
- `我跳下去`
- `我上吊`
- `我現在很危險`

### 第二層：提高警戒，但不一定直接定案
這一層可以先標成高敏感，再決定要不要讓 AI 補做判斷。

例如：
- `最近覺得活著很累`
- `我覺得自己沒必要存在`
- `我真的撐不下去了`
- `我不知道這樣活著有什麼意義`
- `我不想面對明天`

白話理解：
- 這些句子很危險
- 但有些還需要看上下文
- 可以先提高警戒，再決定是直接進 safety，還是 safety-leaning 回應

### 第三層：只是低落，不一定是立即危機
例如：
- `我很累`
- `我很難過`
- `我覺得自己很糟`
- `我最近狀態很差`

這些不應該直接進安全模式。

白話理解：
- 低落不等於立即危機
- 這一層應該交給一般分流，不要太快誤報

### 所以詞庫應該怎麼維護
最合理的方式是把它拆成資料檔，而不是硬寫死在程式裡。

例如可以分成：
- `suicidal_ideation`
- `self_harm_behavior`
- `immediate_danger`
- `passive_death_wish`
- `distress_but_not_critical`

再加上一個欄位：
- `severity = hard / medium / soft`

白話理解：
- `hard`：命中就直接進安全模式
- `medium`：提高警戒，必要時讓 AI 補判斷
- `soft`：只當成情緒低落訊號，不直接進 safety

## 15. 之後還需要用 AI 來切換模式嗎
答案是：**需要，但不需要全部都交給 AI。**

最合理的做法是：

### A. 用規則就能做的，盡量用規則
這些事情其實不用 AI：
- 明確模式指令
  - `auto`
  - `void`
  - `soulmate`
  - `mission`
  - `option`
  - `natural`
  - `clarify`
- 高風險硬命中詞
- 輸出指令
  - `幫我整理給醫生`
  - `請分析我`
  - `產生病人審閱稿`
  - `產生FHIR draft`

白話理解：
- 使用者已經講得很清楚的，不要浪費 AI 去猜

### B. 用 AI 比較有價值的，是灰色地帶
真正值得交給 AI 的，是這些：
- 句子沒有明確說要哪個模式，但語氣顯示互動負擔很高
- 句子不是明確自殺語句，但有被動求死、極度絕望、功能失調的跡象
- 需要分辨 `Soulmate / Mission / Option / Natural` 之間的細微差別

白話理解：
- AI 最有價值的地方不是處理明確規則
- 而是處理模糊、曖昧、需要語境理解的那一塊

### 我認為最好的分工
可以直接這樣理解：

```text
明確指令 -> 規則
明確危險 -> 規則
明確輸出 -> 規則
模糊意圖 -> AI
模糊風險 -> AI
```

### 所以未來的方向不是「全部交給 AI」
而是：
- **規則先擋住明確情況**
- **AI 處理灰色地帶**

這樣的好處是：
- 更快
- 更穩
- 更省 token
- 可預測性更高
- 安全模式不會因為模型一句沒判對就漏掉
