# AI陪伴系統 30 分鐘介紹影片拍攝流程

## 影片目標
- 用一支約 30 分鐘的影片，完整介紹本專案的核心價值、系統操作流程、AI 對話能力、FHIR 資料產出，以及實際送出到 FHIR Server 的驗證結果。
- 影片重點不是只展示畫面，而是要讓觀眾理解：
  - 這套系統解決什麼問題
  - AI 怎麼陪伴使用者並整理資訊
  - 整理後的資料怎麼轉成 FHIR
  - 最後有沒有真的送到 FHIR Server

## 拍攝前準備

### 1. 環境先確認
- 本地伺服器可正常啟動。
- 網頁可開啟：`http://localhost:8787/`
- 健康檢查可用：`http://localhost:8787/health`
- FHIR API 路徑可確認：`http://localhost:8787/api/fhir/bundle`
- 若需要雲端模型，先確認 API Key 已配置完成。
- 若要展示實際上傳，先確認 `FHIR_SERVER_URL` 指向目標測試站。

### 2. 畫面先準備
- 桌面只保留必要視窗，避免通知跳出。
- 建議同時準備這幾個畫面：
  - 瀏覽器中的 AI Companion 首頁
  - 終端機啟動畫面
  - FHIR Server 結果查詢頁面或回傳 JSON
  - 專案簡報頁或 README 摘要

### 3. 對話腳本先準備
- 先準備一組穩定的示範對話，避免現場卡住。
- 建議示範內容涵蓋：
  - 情緒低落
  - 睡眠不佳
  - 壓力來源
  - 日常功能受影響
  - 希望整理給醫師看

## 正式拍攝流程

## 00:00 - 02:00 開場與題目說明

### 畫面建議
- 專案首頁或簡報封面。
- 畫面上顯示專案名稱：`AI Companion`

### 口說重點
- 先說明這支影片要展示什麼。
- 建議口徑：
  - 這是一套用於心理健康診前整理的 AI 陪伴系統。
  - 使用者可以先用自然語言跟 AI 對話。
  - 系統會把對話內容整理成醫療可閱讀的摘要。
  - 最後還能轉成 FHIR 格式，並送到 FHIR Server。

## 02:00 - 05:00 專案定位與系統架構

### 畫面建議
- README 或簡報中的架構說明。
- 可搭配切換到系統主要元件畫面。

### 口說重點
- 說明核心痛點：
  - 病人很難在看診前完整表達狀況。
  - 醫師需要快速掌握重點。
  - 自由文字資料不容易直接交換或驗證。
- 說明系統做法：
  - 前段用 AI 對話接住使用者。
  - 中段整理成可讀摘要。
  - 後段轉成 FHIR / TW Core 導向的結構化資料。

## 05:00 - 08:00 Demo 環境準備

### 畫面建議
- 先切到終端機。
- 展示啟動指令與本地網址。

### 建議操作
```powershell
node app\fhirDeliveryServer.js
```

### 口說重點
- 說明這次示範是在本地啟動。
- 打開：
  - `http://localhost:8787/`
  - `http://localhost:8787/health`
- 補充說明：
  - 這個系統除了對話介面，也包含 FHIR 交付層。
  - 後面會實際驗證資料有沒有送到 FHIR Server。

## 08:00 - 16:00 AI 對話展示

### 示範目標
- 讓觀眾看到這不是死板表單，而是能以自然語言互動的陪伴式系統。

### 建議示範對話
- 使用者先輸入：
  - `最近兩週心情都很低落，晚上也常常睡不好。`
- 第二輪：
  - `白天上班很難專心，下班也沒有力氣做事。`
- 第三輪：
  - `我想先整理一下，之後帶去給醫師看。`

### 拍攝重點
- 停留幾秒讓觀眾看到 AI 回覆內容。
- 口頭指出 AI 做到的事情：
  - 有接住情緒
  - 有追問重要線索
  - 沒有太早跳成僵硬表單
  - 能把自由敘述逐步整理成醫療有用資訊

### 可補拍的第二組情境
- 額外補一小段：
  - `我最近壓力很大，常常凌晨兩三點還睡不著。`
  - `食慾變差，也開始不太想跟人互動。`
- 這段可以用來展示系統對睡眠、食慾、社交退縮等線索的整理能力。

## 16:00 - 21:00 展示輸出成果

### 畫面建議
- 切到系統輸出區。
- 依序展示：
  - 醫師摘要
  - 病人審閱稿
  - FHIR Draft

### 口說重點
- 醫師摘要：
  - 強調這是給臨床端快速閱讀的濃縮版本。
- 病人審閱稿：
  - 強調這份內容較白話，可讓當事人自己確認是否正確。
- FHIR Draft：
  - 強調這已經不是單純文字，而是可交換的結構化資料。

### 建議講法
- 同一段對話可以同時服務三種用途：
  - 給病人自己確認
  - 給醫師快速閱讀
  - 給系統進行標準化交換

## 21:00 - 27:00 實際送到 FHIR Server

### 這段是影片關鍵
- 這一段一定要明確證明：
  - 不是只有做出 JSON
  - 而是真的把資料送出去
  - 並且有伺服器回應或可查詢結果

### 建議展示步驟

#### 1. 先展示 FHIR Draft
- 簡單指出裡面有哪些資源，例如：
  - `Patient`
  - `Observation`
  - `QuestionnaireResponse`
  - `Composition`
  - `DocumentReference`

#### 2. 執行送出
- 在畫面上操作系統的 FHIR 交付流程。
- 若有按鈕就直接按按鈕。
- 若需呼叫 API，也可以展示 API 路徑：
  - `http://localhost:8787/api/fhir/bundle`

#### 3. 展示回傳結果
- 重點拍到以下資訊其中之一：
  - HTTP 成功狀態
  - 伺服器回傳的 Bundle / id / transaction 結果
  - 成功建立資源的回應內容

#### 4. 到 FHIR Server 端確認
- 若目標測試站可查詢，直接打開伺服器頁面或查詢結果。
- 說明：
  - 這一步是為了確認資料不是只停在本地端。
  - 而是確實已經進入 FHIR Server。

### 建議口說重點
- 我們最後不是只停留在 AI 對話。
- 而是把整理後的資料轉成標準格式，並驗證可送入 FHIR Server。
- 這代表這套系統未來比較有機會跟真實醫療資訊流程接軌。

## 27:00 - 30:00 結尾總結

### 建議口說重點
- 總結這次影片展示了三件事：
  - AI 可以先陪伴並協助病人表達
  - 對話內容可以被整理成醫療可讀的摘要
  - 整理後資料可以進一步轉成 FHIR 並送上伺服器
- 最後補一句目前限制與未來方向：
  - 目前仍是輔助整理工具，不是診斷工具
  - 未來可以加強更多 TW Core 對應、真實臨床流程整合與安全治理

## 建議拍攝台詞版本

### 開場版
「大家好，這支影片要介紹的是我們的 AI Companion。這是一套用在心理健康診前整理的 AI 陪伴系統。使用者可以先跟 AI 對話，把近期的情緒、睡眠、壓力與生活影響慢慢說出來，系統再把這些內容整理成醫師摘要、病人審閱稿，以及符合 FHIR 導向的結構化資料，最後我們也會實際驗證資料有沒有成功送到 FHIR Server。」 

### 對話展示過場版
「接下來我會直接示範跟 AI 對話。這裡的重點不是單純聊天，而是讓系統從自然語言中逐步收集醫療上有價值的線索，同時維持陪伴式互動體驗。」

### FHIR 驗證過場版
「接下來我們要看的不是只有畫面上的整理結果，而是實際把這份 FHIR Draft 送出去，並確認 FHIR Server 端是否真的收到資料。」

### 收尾版
「總結來說，這套系統的價值在於把病人的自由敘述、AI 的陪伴互動，以及標準化醫療資料交換串在一起。它不是要取代醫師，而是希望在就醫前先把資訊整理得更清楚，也讓後續系統交換更有依據。」

## 拍攝注意事項
- 若 AI 回覆速度不穩，拍攝前先測一次，避免正式錄製時等待過久。
- FHIR Server 若為公開測試站，可能出現延遲或偶發失敗，建議先做一次預演。
- 若要提升影片穩定度，可先準備：
  - 一次即時展示版本
  - 一次備用截圖版本
- 若實測上傳失敗，不要中斷影片，可直接說明：
  - 本次失敗是測試站連線或回應問題
  - 系統端已完成 FHIR Bundle 組裝與送出流程
