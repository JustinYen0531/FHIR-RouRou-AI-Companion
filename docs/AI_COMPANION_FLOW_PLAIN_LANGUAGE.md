# AI Companion 新版流程圖白話版

## 一句話先講清楚
- 現在的系統不是每一句話都偷偷幫你做完整醫療摘要。
- 現在分成兩層：
  - `聊天層`：先把人接住、先回話、先做必要判斷
  - `輸出層`：只有你真的要摘要、病人稿、FHIR draft 時才生成

這樣做的原因很直接：
- 比較快
- 比較省 token
- 比較不容易撞供應商限流

## 新版整體流程
```text
使用者輸入
  -> 先看是不是模式指令
  -> 再看是不是高風險內容
  -> 再看是不是要輸出醫師摘要 / 病人稿 / FHIR
  -> 如果都不是，就走一般聊天模式
```

## 1. 模式指令流程
如果使用者輸入這些：
- `auto`
- `void`
- `soulmate`
- `mission`
- `option`
- `natural`
- `clarify`

系統不會去跑很重的模型分析，只會：
- 更新模式狀態
- 回一句確認文字

白話理解：
- 這一步像是「切換聊天風格」
- 不是「開始做完整整理」

## 2. 高風險流程
如果使用者說：
- `我想死`
- `不想活了`
- 類似自傷、自殺、立即危險的內容

系統會直接：
- 把這輪標成高風險
- 停止一般模式分流
- 直接走安全回應

白話理解：
- 先救急，不先做摘要
- 先給安全回應，不先做醫師文件

## 3. 一般聊天流程
如果不是模式指令，也不是高風險，也不是輸出指令，就走一般聊天。

這時系統做的是：
- 更新少量重要記憶
- 判斷現在比較適合哪一種聊天模式
- 產生一段回覆

### 一般聊天時會保留的記憶
- 目前模式 `active_mode`
- 是否高風險 `risk_flag`
- 有沒有待追問問題 `pending_question`
- follow-up 次數
- 最近的重要標籤
- 最近的 HAM-D 線索
- 互動負擔程度

白話理解：
- 它會記住「重要線索」
- 但不會每輪都生成「完整報告」

## 4. Follow-up 流程
如果前一輪系統已經留下：
- `pending_question`

那下一輪會先處理這個待追問問題。

白話理解：
- 系統不會忘記剛剛還沒問完的事
- 但追問也有限制，不會無限追著問

## 5. 六個聊天模式現在還在
現在保留的模式有：
- `Void`
- `Soulmate`
- `Mission`
- `Option`
- `Natural`
- `Clarify`

差別在於：
- 以前：很多模式後面會順手帶出一整串摘要流程
- 現在：模式主要只影響「怎麼回話」

白話理解：
- 模式決定聊天風格
- 不再代表每次都要生成文件

## 6. Mission / Option 現在怎麼跑
`Mission` 跟 `Option` 還是有檢索能力，但改成比較節制。

現在的原則是：
- 只有在相關模式下，才用本地知識資料幫助回覆
- 不會在每一次自然聊天都硬做檢索

白話理解：
- 需要知識時才查
- 不需要時不要硬查

## 7. 什麼時候才會產生重型輸出
現在只有兩種情況會跑重型輸出：

### 方法 A：按按鈕
前端按鈕：
- `整理給醫師`
- `病人審閱稿`
- `FHIR Draft`

### 方法 B：輸入明確指令
例如：
- `幫我整理給醫生`
- `產生病人審閱稿`
- `產生FHIR draft`

這時系統才會去生成：
- clinician summary
- patient review
- patient authorization
- FHIR delivery
- delivery readiness

白話理解：
- 平常聊天不自動做
- 你真的要時才做

## 8. 為什麼新版比較快
因為以前最花成本的是這一段：

```text
每輪聊天
  -> tags
  -> burden
  -> hamd
  -> summary draft
  -> clinician summary
  -> patient review
  -> authorization
  -> fhir draft
  -> delivery readiness
```

現在改成：

```text
每輪聊天
  -> 必要判斷
  -> 模式回覆
  -> 保存重要記憶

真的要輸出時
  -> 才跑 summary / patient / fhir 長鏈
```

所以新版比較快，是因為：
- 一般聊天少打很多次模型
- 重型工作延後到真正需要時才做

## 9. 現在你可以怎麼理解這套系統
你可以把它想成兩個角色：

### 角色 1：陪你聊天的人
負責：
- 接住情緒
- 分流模式
- 危險時先做安全回應
- 記住重要線索

### 角色 2：幫你出文件的人
負責：
- 醫師摘要
- 病人審閱稿
- FHIR draft

以前這兩個角色每句話都一起上班。  
現在改成：
- 平常先由聊天的人回
- 你需要時再叫文件的人出來

## 10. 現在最重要的差異
- `聊天` 和 `輸出` 被拆開了
- `記住重點` 和 `生成完整文件` 被拆開了
- `模式切換` 只是切聊天風格，不再等於生成報告
- `醫師摘要 / FHIR` 不是每句都跑，而是按需觸發

## 11. 你之後看畫面時怎麼判斷現在系統在做什麼
如果你只是正常聊天：
- 它應該很快回話
- 不應該每次都很久

如果你按了：
- `整理給醫師`
- `病人審閱稿`
- `FHIR Draft`

那這一輪比較慢是正常的，因為這時系統才會跑重型生成。

## 12. 現在這版的真正目標
- 平常聊天要穩
- 高風險要即時
- 重要記憶不能丟
- 文件輸出要保留
- 不要再因為每輪都跑太多東西，把 API 配額打爆
