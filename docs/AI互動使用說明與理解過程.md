# 與 AI 互動之使用說明、理解過程

本說明用於回應競賽文件中「與 AI 互動之使用說明、理解過程」之要求，整理本專案在近兩週開發期間，如何實際使用 AI 工具協助需求釐清、系統設計、程式開發、測試驗證、文件撰寫與技術除錯。本文強調的是「人與 AI 協作開發」的過程：AI 並非單獨完成作品，而是作為加速分析、實作與重構的工具；最終的需求判斷、品質把關、功能取捨與方向決策，仍由開發者本人持續主導。

---

## 一、整體使用方式

本專案的 AI 使用方式，不是一次輸入一句需求後直接取得最終作品，而是採取多輪式、反覆修正的協作流程。實際工作模式大致如下：

1. 先由我提出需求、描述目標場景與限制條件，例如希望 AI 陪伴系統要能做情緒支持、模式切換、安全分流、FHIR 匯出或競賽展示。
2. 由 AI 協助產生第一版流程、程式碼、文件草稿或架構建議。
3. 我實際執行、匯入、測試、觀察錯誤訊息與介面體驗，再回頭指出問題。
4. 依測試結果要求 AI 進一步修正，例如更改流程路由、補齊欄位、改善 UI、修正相容性、補上 fallback 或強化安全機制。
5. 重複多輪直到功能能穩定執行，並且符合我對實際使用情境、競賽需求與醫療資料格式的要求。

因此，AI 在本專案中的角色，比較接近「可快速協作的技術助理、程式夥伴與文件整理工具」，而不是完全取代開發者。尤其在本專案後期，許多修改已不是單純生成功能，而是根據我連續測試後觀察到的細節問題，再要求 AI 依據實際狀況精準調整。

---

## 二、我如何與 AI 互動並逐步理解系統

在專案初期，我先利用 AI 協助釐清整體產品方向，包含：

- AI 陪伴系統應有哪些主要模式。
- 各模式之間應如何切換與分流。
- 對話內容如何在不同風險程度下有不同處理方式。
- 哪些內容要留在一般陪伴對話，哪些內容應升級為安全流程或臨床摘要。

在這個階段，我不是直接接受 AI 的第一版答案，而是會根據我對心理支持場景、競賽目標與展示需求的理解，持續要求 AI 重寫與重構。從 Git 歷史可以看出，專案在聊天流程上經歷多次調整，例如五模式 routing、void 模式縮限、P0 安全分流、P1 HAM-D 追蹤、P2 負荷狀態、P3 授權與交付流程等，這些都不是一次生成完成，而是反覆修改後逐步成形。

在中期，我開始更理解系統不只是聊天，而是包含狀態管理、風險判斷、摘要草稿、FHIR bundle、前端畫面、資料交付等多層結構。因此，我與 AI 的互動方式也從「請它生成某段內容」轉變為「要求它解釋架構、指出錯誤來源、提出幾種修正方向，再由我選擇哪條路最適合」。這個過程讓我更深入理解：

- 對話流程引擎與 UI 其實是兩條不同但要互相配合的線。
- Dify、Flowise、Node engine 之間的差異，會直接影響流程維護性與相容性。
- 醫療資料不是只要輸出 JSON，而是要對應 FHIR / TW Core 的欄位要求與資源結構。
- 安全模式不能只靠一句 prompt，而需要規則、狀態、流程節點與測試紀錄共同支撐。

也就是說，AI 協助我加速理解系統，但真正讓我掌握整個產品結構的，是我在多輪對話中不斷追問「為什麼這樣設計」、「如果失敗怎麼辦」、「能不能匯入」、「為什麼前端沒有同步」、「為什麼這個欄位不符合 FHIR」等實際問題。

## 三、我在專案中實際使用到的 AI 功能

這次專案中，我使用 AI 的方式非常廣，不只限於單一聊天生成。主要包括以下幾類：

### 1. 程式開發與重構

我使用 AI 協助撰寫與修改前端、流程與後端程式碼，例如：

- 調整聊天流程節點與路由邏輯。
- 重構模式切換與手動 override 指令。
- 製作手機化 SPA 介面。
- 加入聊天畫面的 quick replies、shortcut composer、模式切換區、報表頁與首頁導覽。
- 實作 session persistence、recent sessions、memory view 與 report view。
- 建立 FHIR bundle builder、validator 與 delivery server。

### 2. 除錯與相容性修正

AI 在除錯階段特別有幫助。我會把匯入失敗、欄位錯誤、edge 結構問題、模型設定缺漏、FHIR 上傳衝突等現象交給 AI 分析，再要求它提出修正方案。這部分的實際成果包括：

- 修正 Dify workflow provider 設定。
- 處理 classifier 與 output value type 問題。
- 移除不相容 selector 欄位。
- 修正 orphan edge、skeleton edge structure 與 import sentinel。
- 解決 HAPI FHIR upload collision。
- 補上各種 fallback、debug tracing 與 progress state。

### 3. 文件撰寫與整理

我也大量使用 AI 協助整理產品文件與競賽資料，包括：

- 產品需求文件。
- 白話流程說明。
- 測試檢查清單。
- FHIR 技術摘要。
- Node 引擎說明。
- 競賽實作內容說明。
- 30 分鐘展示流程。
- 專案大事記。

但這些文件不是直接照貼 AI 內容，而是我根據專案實作狀況持續補充、修正與指定語氣，讓它更符合競賽閱讀需求。

### 4. 測試設計與案例補強

我使用 AI 幫忙整理測試項目、手動驗證紀錄與安全情境。例如 mode switching、safety routing、自傷語句 follow-up、FHIR 組成輸出檢查等，AI 可以幫我快速整理測試面向與表格，但實際通過與否、要不要修正，仍由我自己操作系統後判斷。

### 5. 架構討論與方案比較

在系統中後期，我開始把 AI 當成架構討論工具，而不是單純生成器。例如：

- Dify 工作流是否要繼續延伸，還是改成 Node engine。
- Flowise 是否適合作為中間過渡。
- 安全模式是由規則先判還是由模型判。
- 報表、HAM-D、記憶層與 FHIR 交付的資料流要如何串起來。

這些部分，AI 幫我快速提出多種可能方向，但最終採用哪一種，是由我依據競賽時間、可維護性與展示效果做取捨。

## 四、我如何評測、驗證與要求修改

本專案不是「AI 生完就結束」，而是經過大量人工測試與回饋修正。我的評測方式主要有以下幾種：

### 1. 實際匯入與執行

對於 Dify / Flowise / Node engine 相關功能，我會直接匯入流程或執行程式，確認是否能跑、是否報錯、欄位是否完整。只要發現不能匯入、節點不通、欄位不相容，我就要求 AI 重新調整。

### 2. 對話情境測試

我會實際輸入不同類型訊息，測試：

- 正常陪伴對話是否流暢。
- 是否能正確切換模式。
- 低能量、負荷、自傷等高風險內容是否被導到正確路徑。
- HAM-D 與摘要相關訊號是否有被保留。

這也是為什麼 Git 歷史中出現多次 mode routing test、safety routing test、self-harm follow-up record 等紀錄，因為我不是只看程式碼，而是把它當產品來試。

### 3. UI / UX 手動操作

前端部分，我會實際操作頁面，檢查：

- 按鈕是否能點。
- shortcut 與 swipe 是否順手。
- 首頁導覽是否清楚。
- quick replies 是否會干擾輸入。
- scroll、focus、切頁與空狀態是否自然。

因此專案後期出現大量針對 shortcut、layout、spacing、scroll safety、focus behavior 的修正，這些都反映出我實際有做密集操作測試，而不只是接受 AI 給的靜態畫面。

### 4. FHIR 輸出與上傳驗證

在 FHIR 部分，我會確認：

- 資源是否能被正確組 bundle。
- 欄位是否符合預期。
- consent 與 report draft 是否有被正確合併。
- 是否能上傳到 HAPI FHIR。
- 上傳後是否能看到 resource links、Patient ID 與歷史紀錄。

只要流程中斷、欄位異常、上傳衝突或畫面回饋不清楚，我就再要求 AI 修正。

## 五、我使用到的不同 AI 工具與角色分工

這次專案並不是只使用一種 AI，而是搭配不同工具完成不同任務。可大致分為以下幾類：

### 1. 對話型生成與協作工具

我使用像 Codex 這類可直接讀取專案、協助改碼、整理文件與分析 Git 歷史的 AI，作為主要的程式與文件協作夥伴。這類工具的優勢是可以直接根據現有專案上下文進行修改，而不是只提供泛泛建議。

### 2. 模型與對話流程平台

在產品功能層面，我也實際使用到：

- Gemini
- Groq
- Dify
- Flowise

這些工具不是彼此替代，而是在不同階段扮演不同角色。Dify 與 Flowise 偏向流程原型與編排；Gemini / Groq 則是作為實際推理模型或不同路徑的測試選項；Node engine 則是為了提升可控性與整合能力而逐漸形成的自有執行方式。

### 3. AI 作為文件與規格整理工具

除了程式本體，AI 也被我用來將複雜的系統內容轉寫成競賽可讀文件。這包括把技術邏輯改寫成白話說明、把操作流程整理成拍攝 runbook、把 Git 歷史轉成大事記、把測試過程整理成紀錄。這讓我能用更短時間產出完整且一致的專案說明材料。

## 六、這兩週內的技術突破與困境克服

這兩週最大的技術突破，不是單一功能，而是把多條原本分散的線成功接起來。

### 1. 從單純聊天到多模式陪伴系統

最早的聊天流程較接近一般對話型應用，但後來逐步發展為具備模式切換、風險分流、HAM-D 訊號追蹤、治療性記憶與摘要草稿能力的系統。這代表 AI 不再只是「回答問題」，而是開始參與狀態追蹤與支持流程。

### 2. 從工作流原型到程式化引擎

另一個重要突破，是從 Dify 工作流逐步走向 Flowise 過渡與 Node engine 重建。這不只是工具替換，而是表示我已經意識到平台相容性與可維護性的重要性，並透過 AI 協助逐步把系統核心邏輯收回自己可控制的程式碼中。

### 3. 從展示介面到實際互動產品

前端原本只是原型，但經過多輪測試與修正後，加入了首頁導覽、session persistence、recent sessions、報表頁、memory view、mode explainer、shortcut composer 等功能，讓產品更像一個真正可操作、可展示、可評審體驗的系統。

### 4. 從 AI 對話到 FHIR 標準化交付

本專案最具代表性的突破，是成功把 AI 陪伴的輸出接到 FHIR bundle、validation、delivery API 與 HAPI FHIR 上傳流程。這一步相對困難，因為它牽涉到資料結構、欄位映射、錯誤處理、使用者 consent、上傳進度回饋與醫療標準整合。這部分不是 AI 一次生成完成，而是我多次測試、指出問題，再要求 AI 逐步補強。

### 5. 安全模式與高風險內容的處理

最大的困境之一，是如何讓系統在面對自傷、低能量、情緒危機等內容時，不只是回一句安慰，而能進入正確的安全路徑。後來透過規則、路由、測試案例、follow-up 流程與文件整理，才逐步建立出目前較完整的 safety handling 架構。

## 七、我在這個過程中真正做了哪些事情

如果要用最直接的方式說明，我在這個專案中實際做的事情包括：

- 定義產品方向與競賽目標。
- 決定 AI 陪伴系統需要哪些模式、流程與展示重點。
- 持續與 AI 對話，要求它生成、修改、重構與解釋。
- 實際匯入流程、執行程式、操作前端並進行手動測試。
- 根據錯誤與使用體驗，不斷提出新的修改要求。
- 決定哪些 AI 產出要採用、哪些要退回重做。
- 整合 Dify、Flowise、Node engine、Gemini、Groq 與 FHIR 相關技術。
- 把技術成果整理成文件、測試紀錄與競賽敘事。
- 在有限時間內，持續把原型修成可展示、可說明、可交付的版本。

因此，本專案不是單純「請 AI 幫我做一個作品」，而是我主導需求、測試、整合與修正，AI 則在程式撰寫、文件整理、錯誤分析與架構討論上提供高效率協助。真正的開發價值，來自於我如何把 AI 當成工具納入工作流程，並持續用測試、判斷與修正把結果推進到符合競賽需求的程度。

---

## 八、結論

這兩週的開發歷程讓我實際體會到，AI 最有價值的地方不在於一次生成完整答案，而在於它能夠陪著我快速試錯、快速重構、快速整理知識，讓我在有限時間內完成更高複雜度的系統。對我而言，AI 不是替我完成作品的人，而是讓我能更快做到需求分析、原型建構、功能修正、FHIR 整合、文件撰寫與競賽準備的加速器。

如果用一句話總結本專案中的人機協作方式，那就是：我負責定義問題、判斷方向、測試品質與決定取捨，AI 負責加速實作、整理資訊與提出可行方案，兩者反覆協作，才完成目前這個兼具陪伴互動、風險處理、記憶追蹤與 FHIR 交付能力的 AI 陪伴系統。
