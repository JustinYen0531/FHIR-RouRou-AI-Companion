## 影片主線流程

### 1. 用 AI 建立 PHR 病人註冊頁面
影片前段先展示 AI 提示詞與產生結果，重點是請 AI 協助生成一個可上傳 FHIR `Patient` 的網頁。

畫面中可見的設定包括：

- FHIR Server：`https://hapi.fhir.org/baseR4/`
- 程式語言：`HTML + JSON`
- 組織資訊：
  - 組織 id：`53506609`
  - 機構名稱：智慧健康管理中心

AI 生成的頁面用途是：

- 輸入使用者暱稱
- 建立 PHR 端病人
- 將病人資料上傳到 FHIR Server

### 2. 建立病人後回傳 FHIR Patient ID
影片接著示範實際操作註冊頁面。

畫面可見：

- 使用者輸入暱稱
- 點擊上傳至 FHIR Server
- 成功後回傳：
  - `Patient ID`
  - `HTTP Status: 201`

這代表病人資源已成功建立在 FHIR Server 上。

### 3. 建立個人運動紀錄頁面
病人建立完成後，影片進一步展示個人運動紀錄頁面。

從畫面可見，運動紀錄頁面至少包含：

- 目前使用者的 `Patient ID`
- `運動類型 (Activity Type)`
- `運動時間 (Time)`
- 上傳紀錄按鈕

這表示系統已經進入第二階段：

- 不只建立病人基本資料
- 還開始寫入與病人相關的運動事件資料

### 4. 調閱運動紀錄
後段畫面顯示「AI 調閱紀錄」功能，示範如何把病人既有的運動資料查回來並列表呈現。

畫面中可見欄位包含：

- 日期時間
- 運動類型
- 持續時間
- FHIR ID

畫面中的例子有：

- `Running`，`30 min`
- `Swimming`，`60 min`

這代表系統已經具備：

- 從 FHIR Server 查詢紀錄
- 將資料轉回前端表格
- 供使用者檢視或後續更新

## 可推測的 FHIR 實作重點
影片畫面沒有逐段細講所有 resource JSON，但從流程與介面可以合理推測至少涉及下列 FHIR 概念：

### 1. `Patient`
這是影片最明確出現的資源。

用途是：

- 建立 PHR 端病人
- 取得後續所有紀錄關聯所需的 `Patient ID`

### 2. `Observation` 或其他事件型紀錄資源
影片中的運動與生理資料，從 FHIR 建模角度來看，很可能會落在：

- `Observation`
- 或延伸的結構化事件記錄方式

因為畫面中有：

- 運動類型
- 時間
- 持續時間

這些都很適合被表達成某種可觀察、可量化、可追蹤的健康資料。

### 3. 與病人關聯的查詢流程
影片中的調閱頁面明顯依附在病人之下，所以查詢大概率是以病人為主軸進行篩選，例如：

- 指定 `Patient ID`
- 查詢病人相關的運動紀錄
- 將結果回傳前端列表

## 影片呈現的技術價值

### 1. AI 不只是聊天，而是能協助產生 FHIR 前端工具
影片中的 AI 角色不是只負責回答問題，而是進一步協助：

- 生成 HTML 頁面
- 指定 FHIR Server
- 幫忙完成可用的資料上傳介面

這表示 AI 在此脈絡下扮演的是「前端與流程生成助手」。

### 2. FHIR 不只是後端標準，也能進入個人健康紀錄場景
影片不是從醫院 HIS 切入，而是從 PHR 與個人運動紀錄切入。

這很重要，因為它說明：

- FHIR 不只適用於正式病歷交換
- 也適合個人健康資料的標準化整理
- 有利於後續延伸成照護、分析或 AI 摘要

### 3. 具備從建立到查詢的完整閉環
影片不是只展示「可以上傳」，也有展示「可以查回來」。

這代表它已具備最基本的資料閉環：

1. 建立病人
2. 上傳健康紀錄
3. 根據病人查詢紀錄
4. 在前端呈現結果

對展示型專案來說，這比單純展示 JSON 更有說服力。
