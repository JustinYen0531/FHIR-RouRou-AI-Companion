# 決賽導師諮詢問題清單（FHIR 答辯導向）

## 文件定位
- 這份文件是「決賽答辯用」提問稿，不是純知識整理。
- 每一題都必須換到一個可執行決策，避免問完只停在概念。
- 適用前提：目前專案處於「FHIR/TW Core 前結構化準備層」，尚未完成正式 exchange layer。

---

## 諮詢使用說明

### 30 分鐘版（先問高風險 6 題）
1. 先問 `P0-1`、`P0-2`、`P0-3`（資源與映射主軸）。
2. 再問 `P0-4`、`P0-5`（Provenance/Consent 與最小 Bundle）。
3. 最後問 `P0-6`（驗證流程），並請導師給「你決賽當天可以直接講的版本」。

### 60 分鐘版（完整題庫）
1. 先跑完全部 `P0`（8 題），鎖定核心技術決策。
2. 再跑 `P1`（6 題），把命名、版本、流程一致化。
3. 最後跑 `P2`（4 題），準備答辯話術與上線治理說法。

### 問答紀錄模板（每題都填）
| 欄位 | 要填什麼 |
| --- | --- |
| 決策 | 最終採用方案（只能選一個主方案） |
| 理由 | 為什麼選它（臨床、標準、實作、評審角度） |
| 後續動作 | 1~2 個可執行修改（文件/程式/測試） |
| Deadline | 明確日期（YYYY-MM-DD） |

## P0 必問（8 題）

### P0-1 TW Core Profile 最小組合
- 可直接問導師的句子：  
  「以我這個憂鬱症診前整理場景，決賽版最小且合理的 TW Core Profile 組合，你建議一定要鎖哪些？哪些可以延後？」
- 為什麼這題對決賽重要：  
  評審最常追問「你到底有沒有真的對到標準」，這題能先鎖必備範圍。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `Must / Should / Later` 三層清單（至少到 resource/profile 名稱）。
- 拿到答案後你的立即行動（對應專案區塊）：  
  更新 `app/fhirBundleBuilder.js` 的 `TW_CORE_PROFILES` 與 `工程文件入口/AI陪伴系統_FHIR_TWCore整合說明.md` 的最小交付清單。

### P0-2 QuestionnaireResponse vs Observation 切分
- 可直接問導師的句子：  
  「hamd_progress_state 與對話萃取內容，哪些應該放 QuestionnaireResponse，哪些應該轉 Observation，才符合臨床可追溯性？」
- 為什麼這題對決賽重要：  
  這是你資料模型正確性的核心，錯了會被質疑「混放」與「不可驗證」。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `欄位級規則`（例如：原始回答 vs 推定結果）與 2 個反例。
- 拿到答案後你的立即行動（對應專案區塊）：  
  調整 `app/fhirBundleBuilder.js` 的 `collectQuestionnaireItems()` 與 `buildObservationResources()`，同步修訂 `fhir_delivery_draft` 說明。

### P0-3 ClinicalImpression 要不要保留在決賽 MVP
- 可直接問導師的句子：  
  「我的 ClinicalImpression 現在是 preliminary 草稿型，決賽 MVP 保留是加分還是風險？若保留，最低安全欄位應該怎麼定？」
- 為什麼這題對決賽重要：  
  保留可強化臨床印象說服力，但描述不當會被挑戰過度詮釋。
- 期待導師回答到哪個層級（決策格式）：  
  請導師二選一：`保留` 或 `暫緩`，若保留請給 `最低欄位清單`。
- 拿到答案後你的立即行動（對應專案區塊）：  
  依決策調整 `buildClinicalImpressionResource()`、`fhirBundleValidator.js` 規則與競賽口條。

### P0-4 Provenance 與 Consent 邊界
- 可直接問導師的句子：  
  「patient_authorization_state 目前是 workflow state，決賽階段我應該把哪些內容落在 Provenance，哪些保留為 Consent 後續議題？」
- 為什麼這題對決賽重要：  
  評審會直接問授權與責任歸屬，這題能防止你講成『AI 自動送醫囑』。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `現在必做` 與 `下一階段才做` 的邊界表。
- 拿到答案後你的立即行動（對應專案區塊）：  
  更新 `buildProvenanceResource()` 欄位與 `patient_authorization_state` 文件敘述，避免語義重疊。

### P0-5 最小可交付 Bundle 必備欄位
- 可直接問導師的句子：  
  「以 demo/決賽現況，Patient、Encounter、QuestionnaireResponse、Observation、Composition、Provenance 各自最低不可少欄位，你建議怎麼鎖？」
- 為什麼這題對決賽重要：  
  可把『可跑』升級成『可審查』，避免被挑「缺關鍵欄位只是示意 JSON」。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給每個資源至少 3~5 個 `不可少欄位`。
- 拿到答案後你的立即行動（對應專案區塊）：  
  強化 `basicValidation()` 與 `app/fhirBundleValidator.js` 規則，補 `sampleBundleOutput.json`。

### P0-6 驗證流程要到哪一層才夠有說服力
- 可直接問導師的句子：  
  「決賽展示時，我的驗證流程只做本地 validator 夠嗎？還是一定要加上 TW Core profile 驗證或伺服器 round-trip 證據？」
- 為什麼這題對決賽重要：  
  你要證明不是『只會生 JSON』，而是能被標準流程接受。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `最低可接受驗證證據包`（例如：本地規則 + 伺服器回應截圖）。
- 拿到答案後你的立即行動（對應專案區塊）：  
  補齊 `競賽文件入口/FHIR 基礎實作/執行結果.md` 驗證證據段落與 `.logs/fhir-delivery-debug.ndjson` 摘要展示策略。

### P0-7 風險線索該落哪裡最穩
- 可直接問導師的句子：  
  「red_flag_payload 的 warning_tags 與 signals，優先落在 Observation、ClinicalImpression，還是 Composition risk section，怎麼配置最符合審查習慣？」
- 為什麼這題對決賽重要：  
  風險資訊是高敏感區，放錯地方會被質疑臨床語義不清。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `主落點 + 輔落點`，並定義避免重複敘述的規則。
- 拿到答案後你的立即行動（對應專案區塊）：  
  調整 `gatherObservationCandidates()`、`buildClinicalImpressionResource()` 與 `buildCompositionResource()` 的 risk 對位。

### P0-8 交付 gating 與臨床安全敘事
- 可直接問導師的句子：  
  「delivery_readiness_state 與 patient_authorization_state 在決賽口徑上，怎麼講才不被誤解成未授權自動交付？」
- 為什麼這題對決賽重要：  
  這題直接影響你評審信任感與安全邊界說明。
- 期待導師回答到哪個層級（決策格式）：  
  請導師提供 `一句話原則 + 三句說明模板`。
- 拿到答案後你的立即行動（對應專案區塊）：  
  更新 `競賽文件入口/規劃文件/參賽實作內容文件.md` 的交付流程描述與簡報口條。

## P1 建議問（6 題）

### P1-1 Internal Canonical 與版本策略
- 可直接問導師的句子：  
  「我目前用 internal canonical（https://rourou.ai/fhir/internal），決賽階段你建議如何標示版本與治理狀態才專業又不誇大？」
- 為什麼這題對決賽重要：  
  可以避免被質疑「假 canonical」或命名不受控。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `命名規則 + 版本命名格式 + 文件揭露句型`。
- 拿到答案後你的立即行動（對應專案區塊）：  
  更新 `INTERNAL_CANONICALS` 命名註解、`資料規格文件.md` 的版本聲明段落。

### P1-2 CodeSystem 命名與 code 粒度
- 可直接問導師的句子：  
  「我把對話信號寫在 internal CodeSystem（signals），code 粒度應維持症狀層還是風險語意層，才利於未來映射？」
- 為什麼這題對決賽重要：  
  code 粒度會影響可查詢性與後續跨系統對應成本。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `最小 code 粒度原則` 與 3 個建議 code 範例。
- 拿到答案後你的立即行動（對應專案區塊）：  
  調整 `buildObservationResources()` 的 `code.coding` 生成策略與文件範例。

### P1-3 Encounter 粒度定義
- 可直接問導師的句子：  
  「我現在把一次對話 session 對一個 Encounter，若未來同日多段聊天或跨天追蹤，決賽版本先怎麼定義最合理？」
- 為什麼這題對決賽重要：  
  Encounter 粒度不清會讓 longitudinal 解釋混亂。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `決賽版規則` 與 `未來擴充規則` 各 1 條。
- 拿到答案後你的立即行動（對應專案區塊）：  
  更新 `buildEncounterResource()` 註解與 `session.encounterKey` 生成規範文件。

### P1-4 DocumentReference 使用時機
- 可直接問導師的句子：  
  「對我這種診前整理場景，DocumentReference 在決賽該當主資源還是附件型輔助資源，哪種更符合評審期待？」
- 為什麼這題對決賽重要：  
  可避免被認為只存文件、忽略結構化查詢價值。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `主/輔角色判準` 與一個展示順序建議。
- 拿到答案後你的立即行動（對應專案區塊）：  
  調整前端資源顯示順序與 `buildDocumentReferenceResource()` 在輸出說明中的定位。

### P1-5 病人審閱後回寫策略
- 可直接問導師的句子：  
  「patient_review_packet 的 confirm/edit 結果，決賽版應該只回寫到 Composition，還是同步回寫 QuestionnaireResponse/Observation？」
- 為什麼這題對決賽重要：  
  這是『病人參與』是否真的落地的關鍵，不只是 UI 漂亮。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `回寫最小路徑`（只改哪 1~2 類資源）與不建議回寫項。
- 拿到答案後你的立即行動（對應專案區塊）：  
  更新 `patient_review_packet` 映射策略與 `fhir_delivery_draft` 的更新規則說明。

### P1-6 Provenance target 範圍
- 可直接問導師的句子：  
  「我目前 Provenance.target 指到 QuestionnaireResponse、Observation、ClinicalImpression、Composition、DocumentReference，這個範圍會不會過寬？」
- 為什麼這題對決賽重要：  
  target 設計會影響可追溯性可讀性，太寬太窄都會被挑戰。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `必指`、`可選指`、`不建議指` 三層清單。
- 拿到答案後你的立即行動（對應專案區塊）：  
  調整 `buildProvenanceResource()` 的 target 建立策略，並更新測試案例。

## P2 加分問（4 題）

### P2-1 評審答辯話術（30 秒版）
- 可直接問導師的句子：  
  「如果評審只給我 30 秒說明『AI 對話如何安全地轉 FHIR』，你建議我用哪個結構講最穩？」
- 為什麼這題對決賽重要：  
  決賽關鍵在短時間說服，不只在技術正確。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給一段 `30 秒口條模板`。
- 拿到答案後你的立即行動（對應專案區塊）：  
  更新簡報講稿與 `競賽文件入口/規劃文件/參賽實作內容文件.md` 的摘要段。

### P2-2 已知限制怎麼講不扣分
- 可直接問導師的句子：  
  「我目前尚未完成正式 exchange layer，決賽上應該如何誠實揭露限制，同時凸顯完成度與可行性？」
- 為什麼這題對決賽重要：  
  你要避免被認定為 overclaim，又不能把自己講弱。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `可說 / 不可說` 的界線句型。
- 拿到答案後你的立即行動（對應專案區塊）：  
  在 README 與競賽文件中統一「已完成 / 未完成 / 下一步」三段式表述。

### P2-3 上線前法規與治理注意
- 可直接問導師的句子：  
  「若從競賽走向真實場域，這套流程最先會碰到哪三個法規或治理雷點？我決賽應先示警哪些？」
- 為什麼這題對決賽重要：  
  能展現你有醫療資訊治理意識，而不是只做 demo。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `前三名風險 + 最小緩解建議`。
- 拿到答案後你的立即行動（對應專案區塊）：  
  新增「治理風險聲明」段落到工程文件與簡報備援頁。

### P2-4 Roadmap 優先順序
- 可直接問導師的句子：  
  「若我只有 2 週衝刺時間，你建議我的 FHIR 路線優先順序是什麼，才最能同時提升臨床可信度與實作完整度？」
- 為什麼這題對決賽重要：  
  評審常問下一步，這題能把『願景』變『執行路徑』。
- 期待導師回答到哪個層級（決策格式）：  
  請導師給 `Week 1 / Week 2` 任務排序（每週最多 3 件）。
- 拿到答案後你的立即行動（對應專案區塊）：  
  回寫到 `工程文件入口/計畫文件/FHIR 交付一致性與臨床映射補強計畫.md` 的里程碑。

## 決賽前最後確認清單（10 項）
1. 我已用導師答案鎖定 `Must/Should/Later` Profile 清單。
2. QuestionnaireResponse 與 Observation 切分規則已寫成明確條文。
3. ClinicalImpression 已決策 `保留或暫緩`，且理由可 20 秒說清楚。
4. Provenance 與 Consent 邊界已固定，不再混用。
5. Bundle 最小必備欄位已落到 validator 規則，不只是口頭宣稱。
6. 驗證證據包已整理到可展示（規則結果 + 交易/回應證據）。
7. `fhir_delivery_draft`、`delivery_readiness_state`、`patient_authorization_state` 三者關係已可清楚解釋。
8. 決賽簡報與文件中的資源清單說法一致（無互相打架版本）。
9. 已準備「限制揭露話術」與「下一步路線圖」，可直接回答追問。
10. 每一個導師答案都已轉成具體 action 與 deadline，沒有停留在抽象建議。
