# 決賽導師諮詢問題清單

## 這份文件是幹嘛的

你進決賽了，有導師可以問。導師是 FHIR 專家，時間有限，所以要問那些「自己查不到、AI 也不太確定」的問題。以下幫你整理好了，每題都有「怎麼開口問」跟「為什麼要問」。

**建議優先順序：先問一～三，再問四～五，有時間再問六。**

---

## 名詞小字典（看不懂的先看這裡）

| 術語 | 白話意思 |
| --- | --- |
| **LOINC** | 一套國際通用的「醫療項目編號系統」。就像每個商品有條碼一樣，每種檢驗、量表、觀察項目都有一個 LOINC 編號。例如「心情低落」有它專屬的編號，這樣不同醫院看到同一個編號就知道在講同一件事。 |
| **SNOMED CT** | 另一套國際醫療術語系統，比 LOINC 更廣，涵蓋疾病、症狀、身體部位等。可以把它想成「醫療界的共同語言字典」。 |
| **OAuth / OAuth 2.0** | 一種網路授權機制。就是你用 Google 帳號登入其他網站時跳出來問你「要不要允許這個網站存取你的資料」的那個東西。在醫療場景中，用來確保「只有被授權的人或系統才能看到病人資料」。 |
| **互操作性（Interoperability）** | 不同系統之間能互相交換資料、而且看得懂對方的資料。例如 A 醫院的系統產出的資料，B 醫院的系統也能直接讀取和使用，不需要人工重新輸入。FHIR 存在的主要目的就是為了做到這件事。 |
| **Observation** | FHIR 裡的一種資源類型，用來記錄「觀察到的東西」。例如：體溫 37.5 度、病人表示失眠、情緒低落等。我們用它來放 AI 從對話中抓出來的症狀。 |
| **Encounter** | FHIR 裡代表「一次醫療互動」的資源。傳統上是一次看診，我們拿來代表一次 AI 聊天 session。 |
| **QuestionnaireResponse** | FHIR 裡記錄「問答結果」的資源。像是病人填了一份量表，每題的答案就存在這裡面。 |
| **Composition** | FHIR 裡的「文件型資源」，用來把多個資訊組成一份完整的文件。我們拿來裝醫師摘要。 |
| **Provenance** | FHIR 裡記錄「這份資料是誰、在什麼時候、用什麼方式產生的」。我們用它來標記「這是 AI 生成的」。 |
| **Consent** | FHIR 裡記錄「病人同意或不同意某件事」的資源。例如「病人同意把整理好的資料給醫生看」。 |
| **Bundle** | FHIR 裡的「打包」機制。把好幾個資源（Patient、Observation、Composition 等）包成一個 JSON 檔案，方便一次傳送。 |
| **TW Core** | 台灣版的 FHIR 規範。因為每個國家的醫療習慣不同，TW Core 就是「在台灣要怎麼用 FHIR」的指引，規定哪些欄位必填、該用什麼代碼等。 |
| **Profile** | FHIR 的「規格模板」。TW Core Patient 就是一個 profile，它規定了在台灣情境下 Patient 資源應該長什麼樣子、哪些欄位不能少。 |
| **ClinicalImpression** | FHIR 裡記錄「臨床印象」的資源。就是醫生看完病人之後的初步判斷。我們考慮用它來放 AI 的初步分析結果。 |
| **Flag** | FHIR 裡的「警示標記」資源。用來標記需要特別注意的事情，例如「這個病人有自傷風險」。 |
| **RiskAssessment** | FHIR 裡的「風險評估」資源。比 Flag 更詳細，可以記錄風險的程度、依據等。 |
| **ValueSet** | 一組「允許使用的代碼清單」。例如某個欄位只能填「輕度/中度/重度」，這三個選項就組成一個 ValueSet。 |
| **FHIR Server** | 一台專門存放和管理 FHIR 資料的伺服器。可以把 Bundle 送上去，也可以從上面查詢資料。 |
| **dry run** | 「試跑」的意思。模擬一次操作但不會真的執行，用來確認流程沒問題。 |

## 一、我們東西放對地方了嗎？

我們做的事情是：AI 跟病人聊天 → 抓出症狀和風險 → 轉成 FHIR 格式。但很多地方是我們自己猜的，要請導師確認有沒有放錯。

### 問題 1：AI 聊出來的症狀，該放在 FHIR 的哪裡？

我們現在把「失眠」「情緒低落」「自責」這些症狀放在 Observation 裡面，分類設成 survey（篩檢評估）。

**直接這樣問：**
> 我們的 AI 會從聊天裡面抓出病人的症狀，不是醫生判斷的、也不是病人填表填出來的，是 AI 從對話推斷的。放在 Observation、分類用 survey，這樣對嗎？有沒有更好的做法？

**為什麼要問：** 這是整個 FHIR 輸出最基本的東西，放錯的話後面全部都會歪。

---

### 問題 2：「病人跟 AI 聊天」在 FHIR 裡算什麼？

我們用 Encounter（就醫紀錄）來代表一次聊天，但 Encounter 本來是設計給「去醫院看醫生」用的。

**直接這樣問：**
> 病人跟 AI 的對話，我當成一次 Encounter 來記錄。但這不是看醫生，class 這個欄位我不知道該填什麼。填 virtual 可以嗎？還是這種情況根本不該用 Encounter？

**為什麼要問：** 評審如果覺得用法不對，會認為我們不懂 FHIR。

---

### 問題 3：我們的代碼都是自己編的，這樣 OK 嗎？

正式的醫療系統會用 LOINC、SNOMED CT 之類的標準代碼，我們現在大部分都是自己寫中文描述。

**直接這樣問：**
> 我 Observation 的 code 現在是自己寫的文字，沒有用 LOINC。在 TW Core 的標準下這樣會被扣分嗎？如果要補，最少該補哪幾個代碼？可以直接告訴我該用哪個嗎？

**為什麼要問：** LOINC 代碼庫超大，自己查會查到迷路。導師可以直接跟你說該用哪個。

---

### 問題 4：HAM-D 量表的部分跟問答紀錄該怎麼分開放？

我們有 HAM-D 的線索（hamd_progress_state），也有一般的對話整理，但不確定哪些該放 QuestionnaireResponse、哪些該放 Observation。

**直接這樣問：**
> 我系統裡有 HAM-D 量表相關的進度，也有從聊天抓出來的症狀。哪些應該放 QuestionnaireResponse（問答紀錄），哪些放 Observation（觀察結果）？有沒有簡單的判斷原則？

**為什麼要問：** 混在一起放的話，評審會說你「資料模型不清楚」。

---

## 二、AI 生成的東西怎麼讓人信任？

### 問題 5：怎麼標記「這是 AI 寫的，不是醫生寫的」？

我們用 Provenance 來記錄資料來源，想標記：AI 產生 → 病人確認 → 醫生還沒審閱。

**直接這樣問：**
> 我想在 FHIR 裡面標清楚：這份資料是 AI 生成的、病人看過同意了、但醫生還沒看。用 Provenance 來做對嗎？agent 的 type 要填什麼才能代表 AI？

**為什麼要問：** 這是我們專案的賣點之一。Provenance 寫對的話，評審會覺得你有考慮到臨床信任問題。

---

### 問題 6：病人說「可以給醫生看」，要不要建 Consent 資源？

病人在系統裡會按同意，讓整理好的資料可以給醫師。

**直接這樣問：**
> 病人同意把資料給醫生這件事，我需要建一個 FHIR 的 Consent 資源嗎？還是在 Provenance 或其他地方標一下就好？以決賽來說，做到什麼程度就夠了？

**為什麼要問：** 不想做太多也不想做太少，導師能告訴你評審的期望值在哪。

---

## 三、決賽怎麼展示才加分？

### 問題 7：評審最在意 FHIR 的什麼？

**直接這樣問：**
> 以我們這個「AI 對話 + FHIR 輸出」的專案，評審看 FHIR 的部分最在意什麼？是格式合規、能跟其他系統交換、還是臨床上真的有用？我準備 demo 的時候重點該放哪？

**為什麼要問：** 時間有限，力氣要花在評審最看重的地方。

---

### 問題 8：評審如果說「你的 FHIR 只是換個格式而已」，我怎麼回？

**直接這樣問：**
> 如果評審質疑「你們只是把 JSON 換成 FHIR 格式，沒有真正的互操作性」，我該怎麼回應？有沒有什麼角度可以強調 FHIR 在我們場景中的必要性？

**為什麼要問：** 這幾乎一定會被問到，先想好怎麼答。

---

### 問題 9：決賽前如果只能再做一件事，做什麼最有效？

**直接這樣問：**
> 如果我只剩一點時間可以改進 FHIR 的部分，你覺得做什麼最有效果？是把代碼換成正式 LOINC？還是真的送一次 Bundle 到 FHIR Server？還是加上 Consent？

**為什麼要問：** 導師知道什麼最能打動評審，幫你做最有效率的取捨。

---

### 問題 10：驗證只在本地跑夠不夠？

我們有 fhirBundleValidator 做本地驗證，但沒有送到真正的 FHIR Server 過。

**直接這樣問：**
> 決賽 demo 的時候，我只有本地的驗證結果，沒有送到真正 FHIR Server 的紀錄。這樣夠嗎？還是一定要有 Server 那邊回來的驗證結果才有說服力？台灣有沒有公開的測試 FHIR Server 可以讓我試？

**為什麼要問：** 如果能 demo 出「真的送上去了」，比只在本地跑強很多。

---

## 四、高風險訊號怎麼處理？

### 問題 11：病人提到自傷，這在 FHIR 裡怎麼放？

我們會偵測聊天中的高風險訊號（例如自傷想法），目前放在自訂的 red_flag_payload 裡面。

**直接這樣問：**
> AI 偵測到病人可能有自傷風險，這個資訊在 FHIR 裡面該用 Flag、RiskAssessment、還是特別標記的 Observation？在臨床上，醫生通常期望看到什麼格式的風險標記？

**為什麼要問：** 這涉及病人安全，放錯不只是扣分的問題。導師有臨床經驗，知道醫療端怎麼看這種資訊。

---

## 五、我們的限制怎麼講才不會扣分？

### 問題 12：怎麼講「還沒做完」但不會被扣分？

我們的系統還沒接到真正的醫院系統，也沒有完整的 OAuth 授權。

**直接這樣問：**
> 我們目前還沒有接到真正的醫院系統，很多東西是 demo 級的。決賽的時候，怎麼誠實講出這些限制，但又不會讓評審覺得我們做得不夠？有沒有好的講法？

**為什麼要問：** 說太多會扣分，完全不說會被抓到然後更扣分。導師可以教你怎麼拿捏。

---

### 問題 13：交付流程怎麼講才不會被誤會？

我們有 delivery_readiness_state 和 patient_authorization_state 來控制什麼時候可以送資料。

**直接這樣問：**
> 我們的交付流程是：AI 整理 → 病人確認 → 才能送出。但我怕評審誤解成「AI 自動把東西送給醫生、沒人同意就送了」。決賽的時候我該怎麼講這個流程，才能讓評審放心？

**為什麼要問：** 評審很敏感「AI 自動送醫療資料」這件事。講不好會被質疑安全性。

## 問問題的小技巧

1. **帶你的 Bundle JSON 去** — 給導師看你實際產出的東西，他才能給具體建議，不然會聊得很空
2. **先問一～三** — 這些最直接影響決賽分數
3. **導師講的代碼和網址馬上抄下來** — 像 LOINC code、TW Core profile URL，這些你回來 Google 不一定查得到
4. **不要問基礎知識** — 「什麼是 FHIR」「Bundle 是什麼」這種可以自己查的不要浪費導師時間
5. **問完帶回來給 AI 做** — 導師給的方向跟建議，具體的程式碼修改你可以回來找 AI 幫你改
