## 十二、需要優先修正的問題

### P0：自殺意念分數與風險結論矛盾

`QuestionnaireResponse` 中 `suicide` 的 direct value 為 `4`，但 `ClinicalImpression.note` 寫：

```text
No risk signals identified in this session. Requires clinician review to confirm.
```

這是最高優先級問題。即使這只是測試資料，也會讓評審或臨床端覺得系統安全邏輯不可信。

建議修正方向：

- 只要 suicide item > 0，不得輸出 no risk。
- 若 suicide item >= 2，應加入 `clinical_alerts` 或 `safety_flags`。
- 若系統不確定是否為自殺意念，應輸出「requires urgent clinician review」而不是 no risk。

### P0：HAM-D item 有分數，但 total score 為 0

17 個 item 直接加總為 `52`，但：

```text
hamd17_total_ai = 0
severity_band = normal_or_remission
```

這會讓量表結果完全失真。

建議修正方向：

- export 前重新計算 total。
- 若 item 尚未經正式臨床確認，欄位命名改為 `ai_suggested_total_score` 或 `draft_total_score`。
- 若不想自動總分，應將 total 設為 null 或 omitted，而不是 0。

### P1：Export blocker 與 patient authorization 狀態矛盾

Extension 顯示：

```text
patient-review-status = patient_authorized_manual_submit
```

但 Composition 的 `Export Blockers` 仍顯示：

```text
病人需要確認對話內容的正確性
```

建議修正方向：

- 授權後清空 blocker。
- 或把 blocker 改成 historical note：`patient review completed before export`。

### P1：引用了未完整貼出的 Observation

`Observation/3688990` 與 `Observation/3688991` 被多個資源引用，但本次完整 JSON 只包含 `Observation/3688989`。

建議修正方向：

- bundle export 時確保所有 referenced resources 都在同一包。
- 測試紀錄補列每個 Observation 的 code、value、derivedFrom。

### P1：HAM-D evidence 過度重複

許多 item 共用同一段長 evidence，導致不同症狀分數看起來沒有被精準區分。

建議修正方向：

- 每個 item 只保留與該症狀最相關的 evidence。
- Evidence table 內保留短句，full trace 放 DocumentReference。
- 補上 `rating_rationale`，說明分數理由。

### P2：Encounter period 過短

本次 Encounter period 約 3.9 秒，如果代表完整對話，可信度偏弱。

建議修正方向：

- Encounter.period 記錄實際會談期間。
- Export 生成時間另放 Provenance.recorded 或 DocumentReference.date。

### P2：DocumentReference internal trace 過大且敏感

Full internal trace payload 很完整，但包含大量敏感資料與內部推理結構。

建議修正方向：

- 對外交付只放 readable summary。
- internal trace 改為系統內部留存，或在 DocumentReference type 明確標示 internal trace。
- 競賽展示版本去識別化。

## 十三、建議下一版驗收條件

下一次測試建議用以下 checklist 驗收：

- `suicide > 0` 時，不得出現 `No risk signals identified`。
- `hamd17_total_ai` 等於所有 item draft score 加總，或明確為 null。
- `severity_band` 與 total score 一致。
- `patient_authorized_manual_submit` 後，`Export Blockers` 不再要求病人確認。
- 所有被 reference 的 resources 都實際出現在 bundle 中。
- `HAM-D Evidence Table` 每項 evidence 不超過 3 條。
- `rating_rationale` 不為空。
- `DocumentReference` readable summary 與 internal trace 分流。
- 對外文件去識別化。
- `Provenance` 保留生成時間、授權時間與來源 session。

## 十四、可放入競賽簡報的說法

本次測試證明，AI Companion 已能將心理健康對話轉換為一組具備 TW Core 對應、臨床摘要、症狀觀察、病人授權與 Provenance 追溯的 FHIR 交付資源。系統不只輸出文字摘要，也能建立 Patient、Encounter、QuestionnaireResponse、Observation、ClinicalImpression、Composition、DocumentReference 與 Provenance 之間的關聯，使醫師能看到摘要，工程端也能回溯 AI 生成依據。

目前最需要補強的是安全邏輯與量表總分一致性：本次測試中自殺意念 item 被標為高分，但風險摘要仍顯示無風險；HAM-D item 已有分數，但總分仍為 0。這兩點修正後，FHIR delivery 的可信度會明顯提升。

## 十五、結論

這次輸出已經很接近決賽可展示的 FHIR 交付成果：資源種類完整、關聯鏈清楚、病人授權與 AI 生成追溯都有留下來。真正要小心的是，不要讓「看起來完整」掩蓋兩個核心矛盾：風險判斷不能和 suicide item 打架，總分不能和 item scores 打架。

下一版如果把這兩個矛盾修掉，這份 FHIR delivery 就會從「很有架構」前進到「比較能被信任」。這一步很關鍵，也很值得做。
