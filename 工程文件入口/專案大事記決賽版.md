## 一、決賽期的主要變更類型

這段時間的改動，實際上不是只有「新功能」。

1. `新功能`
像是 PHQ-9 雙軌流程、雙角色登入、醫師工作台、Hybrid HAM-D 評分、Clinical Debug Trace。

2. `優化`
像是 FHIR draft 清洗、HAM-D 追問品質、session restore、歷史紀錄保留、FHIR delivery gating。

3. `刪除 / 收斂`
像是把 prompt 裡分散的硬規則移除、避免讓未實作資源混進 FHIR draft、停止直接刪除 legacy session archive。

4. `重構`
像是把症狀整理改成 AI symptom bridge、把臨床控制集中到 `clinicalPostProcessor`、把 auto routing 改成 Smart Hunter 單一出口。

5. `守門 / 穩定化`
像是 auth state normalization、FHIR quick check、question safety、loop detection、restore access 修復。

## V0.9.0 FHIR 草稿與臨床摘要重整期

範圍：`2026-04-08` 至 `2026-04-09`  
代表 commits：`82295f3` → `3bed992`

### 問題（Problem）

- AI 產出的症狀整理容易混進操作句、快捷指令、模式控制語句。
- 病人分析與 clinician summary 容易過度冗長、髒訊息太多，FHIR 草稿可讀性不夠。
- report / draft / bundle 三者之間開始出現宣告與實作不一致。
- 首頁展示與報表結果在重新整理後不夠穩定。

### 核心概念（Key Idea）

- `AI symptom bridge`
- `證據軌 + 推論軌`
- `不信任原始對話，先清洗再映射`
- `FHIR draft 不是展示文案，而是交付前的受控中介層`

### 主要變更

- 新增 AI 症狀對接機制，讓症狀整理從規則摘要主導，改成 AI 語意抽取主導、規則保底。
- 補上非 JSON 輸出 fallback、markdown fallback、防止 false positive、client history hydration、manual report refresh。
- 修整 clinician-focused FHIR report 呈現，降低草稿噪音、保留比較臨床可讀的摘要。
- 對首頁加入可釘選 showcase 對話、output countdown 與 pinned snapshot 穩定保存。
- 修正報表結果刷新後遺失的問題，讓 output 跨 refresh 保留。

### 為什麼這樣改

你這一波明顯是在處理「AI 明明會說話，但不代表它能直接拿去做醫療草稿」這件事。  
所以這版的重點不是再加更多資源種類，而是先把摘要品質、症狀純度、FHIR draft 可讀性拉上來。

### 效果如何

- 這版成功把症狀整理從「關鍵字拼貼」往「臨床摘要」推進。
- 但從 `04-22` 到 `04-24` 又持續大修各 FHIR resource 可看出，這版雖然解了草稿污染問題，還沒有完全解掉「資源契約一致性」問題。
- `04-09` 還補了 refresh persistence，代表報表穩定度一開始確實不夠。

### 取捨（Trade-off）

- 不讓模型直接輸出最終 transaction bundle。
- 不追求華麗報表 UI，先優先處理臨床內容乾淨與可追溯。
- 保留規則式訊號，但降級成 fallback，而不是讓規則直接主導摘要。

## V0.9.1 FHIR 交付守門期

範圍：`2026-04-18`  
代表 commits：`1905704`、`6f54f2c`、`8819530`

### 問題（Problem）

- FHIR delivery readiness 判斷不穩，容易出現「看起來能送，實際上狀態還沒備好」。
- 環境變數來源混亂，runtime key 可能吃到錯的繼承值。
- 缺少快速檢查機制，不容易在 demo 或答辯前立刻確認交付鏈有沒有壞。

### 核心概念（Key Idea）

- `先守門，再交付`
- `readiness normalization`
- `one-click quick check`

### 主要變更

- 正規化 auth / readiness state，補強 FHIR delivery gating。
- 新增一鍵 FHIR delivery quick check。
- 明確偏好 `.env.local` runtime keys，避免被繼承環境覆寫。
- 同步補上決賽差距盤點與導師諮詢文件，開始把交付風險外顯化。

### 為什麼這樣改

這一波不像是擴功能，更像是你發現「現在最危險的不是沒有功能，而是 demo 當下突然送不出去」。  
所以你開始把交付流程做成有守門、有快篩、有環境檢查的系統。

### 效果如何

- 這版有效把 FHIR delivery 從「憑感覺可送」改成「可檢查、可阻擋、可快速驗證」。
- 後面 `04-22` 還繼續修 `ClinicalImpression` status，可見 gating 問題有先穩住，但 bundle 細節仍在補。

## V0.9.2 病人資料接入與 FHIR 資源契約期

範圍：`2026-04-22` 至 `2026-04-24`  
代表 commits：`3de87ee`、`69d1e57`、`6260575`、`38b70e4`、`28d2f35`、`f581b3a`、`e197c47`、`f8091ea`、`68c758d`、`811826b`

### 問題（Problem）

- FHIR draft 缺少足夠的病人 profile 輸入，很多內容只能從聊天硬猜。
- 各資源雖然都在產生，但細節品質參差不齊，存在過度樂觀、重複、狀態不準、證據綁定薄弱等問題。
- FHIR history 雖然有紀錄，但預覽與多筆保存體驗不完整。

### 核心概念（Key Idea）

- `patient profile first`
- `resource-by-resource hardening`
- `單一資源一個一個修，不靠大 prompt 一次包辦`
- `保守輸出勝過華麗亂猜`

### 主要變更

- 把病人 profile intake 接入 FHIR draft，並新增 patient-only refresh flow。
- 支援多筆儲存對話歷史與 FHIR history preview。
- 逐一優化 `Encounter`、`QuestionnaireResponse`、`Observation`、`ClinicalImpression`、`Composition`、`DocumentReference`、`Provenance`。
- 讓 `ClinicalImpression` 真的進 bundle，並讓 bundle status 更貼近 validator 要求。
- 同步產出大量 FHIR 計畫文件、已完成紀錄與理想目標，形成「問題-理想-完成」三分法。

### 為什麼這樣改

你在這一段已經不是在問「能不能產 FHIR」，而是在問「產出來的每一個資源能不能拿得出手」。  
所以策略變成逐資源補強，從欄位值、status、evidence、readability 到 governance 說明全部一筆筆校正。

### 效果如何

- 這版很成功地把 FHIR 交付從單一草稿，推進成有資源契約意識的輸出系統。
- 後續沒有再大規模推翻這批資源修法，表示方向大致成功。
- 但它也暴露另一件事：只修 FHIR 還不夠，因為病人輸入與角色系統不完整時，醫療流程還是會卡，所以後面才接上雙角色與醫師端。

### 取捨（Trade-off）

- 不一次追正式 external canonical 治理。
- 不讓 prompt 自由宣告未實作資源。
- 願意放棄「講得很完整」的草稿，換成比較保守、可驗證、能被 validator 接受的輸出。

## V0.9.3 PHQ-9 雙軌與病人記憶補強期

範圍：`2026-04-23` 至 `2026-04-24`  
代表 commits：`c24ebb4`、`1839ef6`、`a97ca5d`、`fbb5082`、`925cd04`、`84d5349`、`10afdc3`

### 問題（Problem）

- 原本聊天與量表是混在同一條路上，容易互相打架。
- 病人的自評量化資料沒有穩定注入後續對話與 FHIR draft。
- AI 雖然有 therapeutic memory，但缺少更壓縮、更能長期延續的「認識你」記憶層。
- HAMD 報表對逐題分數的可見性不夠。

### 核心概念（Key Idea）

- `雙軌流程`
- `量表獨立，但要能回灌對話`
- `compressed know-you memory`
- `自評資料不是附件，而是後續理解病人的一部分`

### 主要變更

- 新增 PHQ-9 獨立入口、逐題分數與自由敘述、版本保存與最新值注入。
- 把 PHQ-9 接進 structured drafts 與 FHIR 內容。
- 新增 compressed know-you memory 與一鍵 memory compression test。
- 報表中顯示 HAM-D item score details。

### 為什麼這樣改

這一版很明顯是在解「聊天很會聊，但量表資料沒有真正被系統吃進去」的問題。  
你不是要把 PHQ-9 當第二個聊天頁，而是要讓它成為和自然對話平行、卻能互相補強的醫療資料軌。

### 效果如何

- PHQ-9 雙軌方向後面沒有被砍掉，代表這條設計基本成立。
- `a97ca5d` 補 prompt gating、button state，表示第一版 UI / 流程仍有互動瑕疵，但不是方向錯，是收邊還不夠。
- 後面醫師端也開始讀取病人自評摘要，表示這版成功成為後續雙角色協作的資料基底。

### 取捨（Trade-off）

- 不讓 PHQ-9 取代 HAM-D。
- 不把量表流程直接塞回一般聊天畫面。
- 不在這版自動做診斷結論，只做整理、注入與輸出。

## V0.9.4 雙角色登入與醫師工作台期

範圍：`2026-04-24` 至 `2026-04-25`  
代表 commits：`cd9a4fa` → `0d41654`

### 問題（Problem）

- 系統原本偏病人單機整理工具，缺少正式的病人 / 醫師角色分流。
- 醫師沒有自己的入口，病人資料也沒有穩定的 account / patient ID 綁定。
- 初版 auth 一上線後，立刻遇到 first login、state stale、session migration、report reload、assignment persistence 等一整串現實問題。

### 核心概念（Key Idea）

- `role-based entry`
- `doctor workspace before full HIS integration`
- `先把誰是誰、誰看誰、誰能指派誰建立起來`
- `prototype first, hospital-grade later`

### 主要變更

- 建立 v1 dual-role authentication system。
- 將登入改成 modal + API auth routes，補強 register / me / logout / users 流程。
- 新增醫師端病人管理工作台 prototype、病人列表、病人詳情、FHIR-mapped medical record input、doctor order form。
- 讓醫師可用病人 account id 加入病人，並在後端保存 assignment。
- 補上 patient self-report summary 給醫師讀、顯示 attending doctor、persist patient reports and PHQ9 state。
- 一路修正 first login、stale auth、expired prompt、shared history、ephemeral 訊息過濾、登入時 legacy session migration、report cache restore。

### 為什麼這樣改

從這串 commit 看得出來，你的系統已經不想只停在「病人自己聊天整理」。  
你要的是一條真正能被醫師接手的流程，所以先從最原始但最必要的事情開始：角色、登入、病人 ID、工作台、指派、摘要閱讀。

### 效果如何

- 這版方向非常關鍵，因為它讓整個系統從單人 prototype 變成醫病雙角色 prototype。
- 但它也是決賽期最典型的「功能一開，修補洪流立刻湧來」案例。
- `04-25` 同一天的密集修正，說明第一版 auth/workspace 架構是對的，但邊界條件、暫存狀態、帳號遷移和資料一致性一開始非常脆弱。

### 取捨（Trade-off）

- 不直接接 HIS / EMR。
- 不做正式院內 SSO。
- 不讓醫師端進病人聊天介面。

## V0.9.5 HAM-D 訪談骨架與自動分流重構期

範圍：`2026-04-26`  
代表 commits：`743bdcd`、`c96240f`、`85a7d9c`、`e27a908`

### 問題（Problem）

- mission/clinical probing 太像「有同理但不夠可評分」的聊天。
- HAM-D 進度只顯示百分比，不知道缺哪題、缺什麼。
- Auto 模式的路由太機械，會亂跳風格，也會在錯的時機插入探針。
- 對話自動存檔與歷史保留不夠穩。

### 核心概念（Key Idea）

- `三段式臨床訪談`
- `可評分優先（Scoreable First）`
- `Smart Hunter 單一出口`
- `先保護氣氛，再決定是否進 probe`

### 主要變更

- 把 HAM-D 評估改成 11 題核心項目、逐題 complete / partial / missing 的缺口模型。
- 前端新增逐題缺口指示器 UI，不再只有 completion 百分比。
- Auto routing 從外部多 handler 切換，改成 Smart Hunter 單一出口、內部子模式切換。
- 新增信號導向、半步切換、`flow_state` 追蹤。
- 修正歷史紀錄過濾邏輯與自動存檔，避免對話遺失。

### 為什麼這樣改

這版很像你正式承認一件事：  
「如果 AI 要做 HAM-D，它不能只會聊，還要會蒐證，而且蒐證進度要被看見。」

### 效果如何

- 這版替後面所有臨床控制打下骨架，尤其是缺口指示器與單一出口 routing。
- 但 `04-27`、`04-28`、`04-29` 還一路補 question safety、換題、分類、trace，表示骨架立起來之後，真正困難的是把 LLM 每一輪輸出鎖進這個骨架裡。

- UI 不做 fancy 可視覺化圖表，先把缺口、證據、建議追問講清楚。

## V0.9.6 臨床後處理器與決策可視化期

範圍：`2026-04-27`  
代表 commits：`79b0809` → `3f27e72`

### 問題（Problem）

- LLM 會自己長出不該問的問題，例如泛問、安慰式問句、coping 問句、錯 item 問句。
- 風險訊號容易過度敏感或不夠可解釋。
- 當 AI 問得怪時，外部很難知道是 prompt 問題、規則問題，還是後處理問題。
- 加入新規則後，如果沒有統一控制點，很快就會散得到處都是 patch。

### 核心概念（Key Idea）

- `LLM 自由生成，後處理器唯一守門`
- `clinicalPostProcessor 單一控制點`
- `Debug trace 可視化`
- `系統閉嘴條件`

### 主要變更

- 建立 `completion lock`、`Question Type Lock`、`enforceQuestionSafety`、`Scoreable Question Enforcement`。
- 一度加入 `Scoring Breakpoint`，又撤銷，再改成小括號附註重新加入，形成一次真實的試錯循環。
- 把 prompt 中的硬規則抽掉，改由 engine 後處理器統一守門。
- 建立 `clinicalPostProcessor()`，收斂成單一控制點，整合風險優先、問句替換、單問句、wrong item 介入與 crash fallback。
- 新增 Clinical Debug Trace、AI 決策紀錄面板、白話 trace 文案與 debug 按鈕。
- 補上風險 trace 欄位、trace 空狀態、風險冷靜機制與更完整的 override 規則。

### 為什麼這樣改

你這版的設計思想很鮮明：  
不是要求 prompt 永遠乖，而是承認 LLM 會飄，所以最後一定要有一個 deterministic 的關卡把它攔下來。

### 效果如何

- 這版是整個決賽期最重要的工程重構之一，因為之後所有 04/28、04/29 的改動幾乎都建立在它上面。
- `Scoring Breakpoint` 先加又撤，證明你不是一路順風，而是真的測到副作用才重做。
- 後面還持續補 switch 邏輯、clarify mode、semantic dimension，說明單一後處理器成功了，但 target selection 還需要更上層的語意判斷。

### 取捨（Trade-off）

- 刪掉 prompt 內大量硬限制，改由程式接管最後控制。
- 接受系統更「管很多」，換取可評分、可追蹤、可 debug。
- 不追求 AI 完全自由，而是追求自由之後仍能被矯正。

## V0.9.7 Hybrid HAM-D 評分與防繞圈期

範圍：`2026-04-28`  
代表 commits：`c46c118` → `d8f5743`

### 問題（Problem）

- 只靠 NLP 自動判分，仍然會遇到病人其實心裡有尺度，但語句沒有完全講出來。
- HAM-D 容易卡在同一題繞圈。
- wrong item 介入有時太硬，會把本來自然跟進的問句硬切掉。
- 決策面板雖然出來了，但互動與視覺仍需要更可讀。

### 核心概念（Key Idea）

- `Hybrid scoring`
- `Human-in-the-loop`
- `probe_meta`
- `循環偵測與強制換題`

### 主要變更

- 加入 slider UI、`user_self_rating`、`pending_probe_meta`，把使用者自評納入 HAM-D 評分。
- 評分改為 `0.7 NLP + 0.3 user` 的混合權重。
- 連續修改 HAM-D 換題邏輯，加入 Change 1+2、Change 4+5。
- 補上 wrong_item 豁免、17 題關鍵詞擴充、繞圈偵測、過多探針暫排除。
- 同步調整決策面板與滑桿 UI，讓臨床互動不那麼生硬。

### 為什麼這樣改

你這版其實是在解一個很聰明但很現實的問題：  
「如果 AI 問得不夠準，那就讓病人自己用滑桿補上一點結構化訊號。」  
這不是把決定權完全交給使用者，而是把 AI 判斷和人類主觀感受混成一個更穩的評分來源。

### 效果如何

- Hybrid scoring 是明顯的新能力，後面 `04-29` 還繼續沿用，表示方向成功。
- 但同一天就連續兩版換題邏輯，隔天又繼續補「問一次就換題」「有滑桿+文字證據就直接換題」，這代表最難的不是評分本身，而是換題節奏。

### 取捨（Trade-off）

- 不再迷信純 AI 判分。
- 接受 UI 變得更像半結構化訪談，而不是純陪伴聊天。
- 沒把 slider 做成每輪都強迫填，保留略過與文字證據共同作用。

## V0.9.8 LLM 語意換題與 restore 體驗整合期

範圍：`2026-04-29`  
代表 commits：`086803b` → `0e6be17`

### 問題（Problem）

- 換題邏輯如果只看關鍵字或 pending lock，還是不夠語意化。
- clarify / probing / flow 三種狀態邊界容易互相打架。
- session restore 後，trace 按鈕、歷史紀錄、output refresh 會出現斷裂。
- 同題重問、過早 clarifying、建議式問句等殘留問題仍會破壞訪談品質。

### 核心概念（Key Idea）

- `conversation_mode 3-state`
- `LLM semantic dimension classification`
- `主軸優先，不亂切 clarifying`
- `換題要看語意與證據，不只看關鍵字`

### 主要變更

- 將 `conversation_mode` 三態正式整合進 `clinicalPostProcessor`，並可在 trace 中顯示。
- 新增 LLM conversation-mode trace、decision source trace、flowchart path、LLM vs rule 視覺化。
- 讓 probing mode 真正驅動 HAMD item reuse。
- 改用 LLM semantic dominant dimension 來決定 target item，而不是單純 keyword count。
- 加入 dimension switching transition sentences。
- 禁止主軸仍在時切入 clarifying，clarify mode 也禁止跑 slider prompts。
- 有滑桿 + 文字證據時直接換題；最後進一步收斂成「問一次就換題」，並大幅提高高風險觸發門檻。
- 修復 restore access、restore progress、recent history 載入、home feedback、trace button 與跨頁按鈕常駐。

### 為什麼這樣改

這版看起來像最後衝刺，但其實是在處理最本質的東西：  
不是「有沒有規則」，而是「規則到底要在什麼語意狀態下出手」。  
你開始讓系統理解對話主軸、情緒澄清、正式追問之間的差別，而不是把所有問句都丟進同一個決策桶裡。

### 效果如何

- 這版成功把決策邏輯從「規則堆疊」再往前推到「語意狀態機 + trace 可解釋」。
- 但從同一天大量連續 commit 看得出來，這是高密度調參區，代表你是在實戰對話中邊測邊修。
- 最後收斂到「問一次就換題」與高風險門檻拉高，表示你最終選擇的是寧可保守，也不要讓系統一直黏著同一題打轉。

### 取捨（Trade-off）

- 放棄只靠 keyword count 的簡單換題法。
- 不讓 clarifying mode 同時承擔正式評分任務。
- 願意犧牲一點探針深度，換取節奏穩定與不重問。
