# 決賽可交付差距盤點與必加項目

更新日期：`2026-04-18`

## 一句話結論（你現在離決賽版多遠）
- 目前整體完成度可視為：**約 72 / 100**（已可展示、可跑通、核心測試全綠）。
- 距離「決賽可交付且可被評審信任」版本，還差：**約 28 / 100 的收尾工作**。
- 這 28 分不是在核心功能，而是在：**可驗證證據、交付敘事一致性、授權與風險邊界說明**。

---

## 現況盤點（已具備 vs 主要缺口）

### 已具備
1. FHIR 核心輸出鏈已成形：`Patient / Encounter / QuestionnaireResponse / Observation / ClinicalImpression / Composition / DocumentReference / Provenance`。
2. `delivery_readiness_state` 與 `patient_authorization_state` 已有 gating，非授權狀態可阻擋輸出。
3. 本地自動測試全綠（`node --test app\\*.test.js`，共 5 個測試檔皆通過）。
4. 具備 dry-run 與 transaction 路徑，並可對公開 HAPI target 做唯一 key 處理。

### 主要缺口
1. 缺「決賽證據包」：目前缺少一組可直接給評審看的完整交付證據（真實交易回應 + 驗證報告 + 關鍵欄位對照）。
2. 文件說服力不一致：`競賽文件入口/FHIR 基礎實作/程式碼.md` 內容偏薄，尚不足以支撐技術答辯深度。
3. 授權與來源追溯仍偏流程描述：有 state，但缺「可展示的最小治理證據格式」。
4. 高風險訊號落點與口徑尚未固定：答辯時容易被追問「風險資料到底落在哪裡、誰負責判讀」。

---

## 必加項目（P0，決賽前一定要補）

> 原則：以下項目是「不補會直接影響決賽評分與信任度」的最低必做清單。

### P0-1 決賽證據包（非口頭版）
- 為什麼一定要加：  
  評審要看的不是你說可以，而是你有證據顯示「真的可交換、可驗證」。
- 完成標準：  
  1) 至少一組 `bundle_json` 範例（含 resource index）  
  2) 至少一組 transaction response（成功或可解釋失敗）  
  3) 至少一份 validation 結果摘要（error/warning/issue_count）  
  4) 以上三者可互相對照同一個 encounterKey
- 對應專案區塊：  
  `app/fhirDeliveryServer.js`、`app/fhirBundleValidator.js`、`競賽文件入口/FHIR 基礎實作/執行結果.md`

### P0-2 最小交付契約凍結（欄位不可再漂移）
- 為什麼一定要加：  
  沒有凍結欄位，決賽前每次改 prompt 或 merge 都可能讓輸出漂移。
- 完成標準：  
  1) 明確列出每個 resource 的最低必備欄位  
  2) `fhir_delivery_draft.resources` 與實際 builder 輸出完全一致  
  3) validator 規則與文件說法一致
- 對應專案區塊：  
  `app/fhirBundleBuilder.js`、`app/fhirBundleValidator.js`、`工程文件入口/AI陪伴系統_FHIR_TWCore整合說明.md`

### P0-3 授權與追溯最小治理證據
- 為什麼一定要加：  
  你作品是醫療資訊場景，沒有授權與來源邊界，評審會直接扣信任分。
- 完成標準：  
  1) `patient_authorization_state` 至少補齊：同意狀態、同意範圍、時間點、操作者  
  2) Provenance 明確標示 AI 產出與病人審閱狀態  
  3) 文件中有一段「未同意時不交付」的固定口徑
- 對應專案區塊：  
  `app/fhirBundleBuilder.js`（Provenance/extension）、`競賽文件入口/規劃文件/參賽實作內容文件.md`

### P0-4 風險訊號固定落點與安全口徑
- 為什麼一定要加：  
  自傷/高風險問題是決賽高機率追問點，不能現場臨時解釋。
- 完成標準：  
  1) 明確定義 `red_flag_payload` 主落點（Observation/ClinicalImpression/Composition 哪個為主）  
  2) 明確定義「AI 只做線索整理，不做診斷」固定句  
  3) 文件與 Demo 話術一致
- 對應專案區塊：  
  `app/fhirBundleBuilder.js`、`工程文件入口/AI陪伴系統_FHIR_TWCore整合說明.md`、簡報講稿

### P0-5 競賽文件的技術深度補強（特別是程式碼頁）
- 為什麼一定要加：  
  目前程式碼頁偏連結導向，評審看不到「你到底實作了什麼判斷與驗證」。
- 完成標準：  
  1) `競賽文件入口/FHIR 基礎實作/程式碼.md` 增加：核心模組職責、輸出流程、關鍵測試結果  
  2) 直接列出 1 段範例資源與對應 state 來源  
  3) 補上「已完成 / 未完成 / 下一步」三段式結論
- 對應專案區塊：  
  `競賽文件入口/FHIR 基礎實作/程式碼.md`、`競賽文件入口/FHIR 基礎實作/說明文件.md`

### P0-6 決賽答辯與 Demo 固定腳本
- 為什麼一定要加：  
  技術有了但口條不固定，現場很容易答到互相矛盾。
- 完成標準：  
  1) 30 秒版本定位句（做什麼、不是做什麼）  
  2) 90 秒版本流程句（對話→結構化→FHIR→授權→交付）  
  3) 3 個高機率追問的固定答法（互操作性、臨床責任、限制揭露）
- 對應專案區塊：  
  `工程文件入口/決賽導師諮詢問題清單.md`、競賽簡報稿
